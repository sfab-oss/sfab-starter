# ALW-334 Remove unused and stale comments from the codebase

## Context

`sfab-starter` accumulated comments during extraction from `sfab` and early scaffolding — leftover TODOs, commented-out code blocks, narrating-the-obvious comments, and notes that no longer match the code. These add noise for humans and agents reading the template and make it harder to spot comments that carry real intent (non-obvious business rules, ADR rationale, sharp edges).

The project already follows a lean-comment convention in `AGENTS.md` (comments should explain *why*, not *what*). This task brings the repo in line with that standard.

## Scope

- Audit the repo for comments that are unused, stale, or redundant and remove them.
- Remove commented-out code that is not intentionally preserved (e.g. no active ADR/plan reference).
- Remove narrating comments that restate what the code already says.
- Keep comments that explain non-obvious business logic, security constraints, platform sharp edges, or decision rationale.
- Do **not** strip JSDoc/TSDoc on exported public APIs unless it is empty or purely redundant.
- Do **not** edit generated files, vendored assets, or lockfiles.

## Acceptance criteria

- [ ] AC-1: No commented-out executable code blocks remain unless tied to an in-repo ADR/plan with an explicit reason. verify: `rg '//.*\b(const|let|var|function|import|export|return|if|for|while)\b' --glob '!**/node_modules/**' --glob '!**/*.lock'` returns only justified cases
- [ ] AC-2: Obvious narrating comments (e.g. `// increment counter` above `counter++`) are removed across `apps/`, `packages/`, and `docs/` where they add no information. verify: spot-check changed files; no new narrating comments introduced
- [ ] AC-3: Stale TODO/FIXME/HACK comments that no longer apply are removed or updated to reflect current truth. verify: `rg 'TODO|FIXME|HACK' --glob '!**/node_modules/**'` — each remaining hit is still accurate
- [ ] AC-4: `pnpm lint`, `pnpm typecheck`, and `pnpm test` pass with no behaviour changes. verify: run all three from repo root
- [ ] AC-5: Valuable explanatory comments (business rules, non-obvious constraints, ADR cross-refs) are preserved. verify: PR description lists any borderline keeps with rationale

## Non-goals

- Rewriting or expanding documentation in `docs/` beyond removing stale inline commentary.
- Refactoring code for clarity (rename variables, extract functions) — comment removal only.
- Adding Biome or ESLint rules to enforce comment policy (follow-up if desired).
- Touching `.sfab/` mirror files by hand.

## Boundaries

- ✅ Always: preserve comments that explain *why*; keep exported API docs that agents/humans rely on; run lint/typecheck/test before merge.
- ⚠️ Ask first: removing a comment you cannot verify is stale; deleting large commented blocks that might encode a pending design.
- 🚫 Never: change runtime behaviour; strip license headers or copyright notices; edit generated/vendor output.
