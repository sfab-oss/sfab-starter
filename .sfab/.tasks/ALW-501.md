# ALW-501 URL-backed table/view state + mirror into page context

## Context

Page-context for the org agent is intentionally lightweight (entity type/id/title → system prompt). List views already keep pagination/sort/search in the URL for React Query, but that view fingerprint is **not** mirrored into chat page context — so the agent cannot reproduce "what you're looking at." Other UI state may also belong in the URL (tabs, dialogs, filters) and should be audited.

## Scope

- Audit starter tables (catalog, entities, documents, invitations, settings lists) for URL-backed filters/sort/pagination vs local-only state.
- Audit other stateful UI where URL params make sense (tabs, selected filters, search).
- Fix gaps so list views that matter for agent context expose their search params; optionally extend page-context publish to include those params (lightweight — not row payloads).
- Document the convention: URL owns shareable view state; page context mirrors enough for the agent to re-query the same view.

## Acceptance criteria

- [ ] AC-1: Inventory of list/table routes with URL vs local state (artifact or task comment)
- [ ] AC-2: Catalog/entities/documents list filters that affect the visible query are in the URL (or explicitly documented as out of scope)
- [ ] AC-3: Page context for list routes includes the view fingerprint (search/sort/page or equivalent) so the agent can call the same list tool — verify: system-prompt section on a filtered catalog
- [ ] AC-4: No full row payloads in context; IDs + query params only

## Non-goals

- Redesigning page-context chip UX; agent tool invalidation (separate task).

## Boundaries

- ✅ Always: keep context lightweight; prefer URL as source of truth for shareable view state.
- ⚠️ Ask first: changing default pagination sizes or public URL shapes that break bookmarks.
- 🚫 Never: dump full table rows into the system prompt.
