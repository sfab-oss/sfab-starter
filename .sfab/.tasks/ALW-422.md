# ALW-422 Starter QA sweep — dogfood verification of implemented surfaces (Group)

## Context

The starter now has a real ERP surface (auth/orgs, catalog, documents + payments, settings/RBAC, org-agent chat, global shell) and the first dogfood QA pass (2026-07-08, `starter-dogfood-qa` workspace packet) produced [[ALW-419]] and [[ALW-421]], both merged. This Group organizes the systematic follow-up: one QA session per area, covering both **UX quality** (best-practice look/feel, responsive, accessibility) and **functional correctness** (does what we implemented actually work).

Method per child task: run the app locally, drive it with browser QA (desktop + mobile viewports), capture screenshot/video evidence, log findings in the session tracker, and file confirmed defects/polish as draft fix tasks — QA tasks do not fix, they verify and report.

## Scope

Children (one QA session each):
- Shell, navigation & command-K palette
- Auth, onboarding & org lifecycle (incl. RBAC gating)
- Catalog end-to-end
- Documents & money loop (invoice → finalize → payment → allocations)
- Org-agent chat & approvals wiring
- Cross-cutting UX (theming, states, responsive, a11y, i18n)

## Non-goals

- Fixing findings inside QA sessions (each confirmed finding becomes its own draft task).
- `apps/docs` and the component-registry build tooling (`packages/registry`) — not part of the app runtime surface.
