# ALW-426 QA: documents & money loop — invoice → finalize → payment → allocations

## Context

This is the functional heart of the ERP base: Transaction Core settlement ([[ALW-354]]) and the credit wallet ([[ALW-355]]) merged with heavy oversight, but the UI on top has only had a light dogfood pass and the tracker calls the invoice flow out as weak (B-8, deferred redesign). The surface map also showed the UI under-exposes the core; that build-out is now planned as group [[ALW-432]] ([[ALW-433]] entities, [[ALW-434]] wallet, [[ALW-435]] document types), so this QA session verifies what exists today and feeds those tasks rather than re-deciding scope.

## Scope

- Invoice lifecycle: create draft, add/remove line items, live totals, finalize, then verify a finalized doc is immutable where it should be.
- Money correctness through the UI: subtotal/tax/total math vs entered minor-unit/bps values; record payment (cash/transfer/card); partial payment; overpayment attempt; payment-status badge transitions; balance-due behavior; allocation correctness against the API.
- Activity feed accuracy on the hub and the document detail.
- API-level spot checks of the surfaces that have no UI yet (wallet deposit/redeem conservation, entities, payment reversal) so the [[ALW-432]] tasks start from verified behavior.

## Acceptance criteria

- [ ] AC-1: Full invoice→finalize→payment loop verified on desktop + mobile with evidence, including partial and attempted over-payment.
- [ ] AC-2: Displayed totals cross-checked against API responses for at least 3 documents (verify: compare UI numbers to `documents`/`payments` endpoint payloads).
- [ ] AC-3: Immutability of finalized documents probed through the UI and reported.
- [ ] AC-4: API-only surfaces (wallet, entities, payment reverse) spot-checked over HTTP with findings recorded on the corresponding [[ALW-432]] child tasks.
- [ ] AC-5: Findings logged; B-8 material handed to [[ALW-430]] rather than re-filed.

## Non-goals

- Building the missing UI (group [[ALW-432]]) or the invoice UX redesign ([[ALW-430]]).

## Boundaries

- ✅ Always: seeded local DB; evidence per finding.
- ⚠️ Ask first: schema-level probing beyond the HTTP API.
- 🚫 Never: code changes on this branch; prod data.
