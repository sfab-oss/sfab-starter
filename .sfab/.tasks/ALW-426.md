# ALW-426 QA: documents & money loop — invoice → finalize → payment → allocations

## Context

This is the functional heart of the ERP base: Transaction Core settlement ([[ALW-354]]) and the credit wallet ([[ALW-355]]) merged with heavy oversight, but the UI on top has only had a light dogfood pass and the tracker calls the invoice flow out as weak (B-8, deferred redesign). The surface map also shows the UI under-exposes the core: only sales invoices are creatable although the backend supports more document types/directions; wallet (`hono/org-protected/wallet.ts`) and entities (`entities.ts`) have full APIs and **zero UI**; line-item and payment forms take raw minor-units / basis-points.

## Scope

- Invoice lifecycle: create draft, add/remove line items, live totals, finalize, then verify a finalized doc is immutable where it should be.
- Money correctness through the UI: subtotal/tax/total math vs entered minor-unit/bps values; record payment (cash/transfer/card); partial payment; overpayment attempt; payment-status badge transitions; balance-due behavior; allocation correctness against the API.
- Activity feed accuracy on the hub and the document detail.
- Gap audit (report, don't build): document types/directions unreachable from UI; wallet deposit/redeem/balance with no surface; entity management with no surface ("Walk-in" only).

## Acceptance criteria

- [ ] AC-1: Full invoice→finalize→payment loop verified on desktop + mobile with evidence, including partial and attempted over-payment.
- [ ] AC-2: Displayed totals cross-checked against API responses for at least 3 documents (verify: compare UI numbers to `documents`/`payments` endpoint payloads).
- [ ] AC-3: Immutability of finalized documents probed through the UI and reported.
- [ ] AC-4: Written gap report with a recommendation per gap (expose in UI / keep agent-only / drop): doc types, wallet, entities.
- [ ] AC-5: Findings logged; B-8 material handed to the invoice-redesign task rather than re-filed.

## Non-goals

- The invoice UX redesign itself (separate task).
- Building wallet/entities UI (the gap report decides whether tasks get filed).

## Boundaries

- ✅ Always: seeded local DB; evidence per finding.
- ⚠️ Ask first: schema-level probing beyond the HTTP API.
- 🚫 Never: code changes on this branch; prod data.
