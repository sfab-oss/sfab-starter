# ALW-526 Comment hygiene: docs + REMOVE-only scrub

## Context

Code should be self-descriptive. Comments earn their place only when they carry information the code cannot express (business process, external constraints, races, non-obvious why). Narration, section banners, name-echo JSDoc, and step labels that could be replaced by cleaner structure are noise. Prior cleanups ([[ALW-334]], platform [[ALW-338]]) never left a durable rule in starter `AGENTS.md`.

Local audit (2026-07-10): starter is not comment-bloated (~5–14% trim candidates). Hotspots: transaction batch assemblers (banners / numbered+lettered steps), a few agent JSDocs, JSX banners in `app-sidebar`.

## Scope

- Add short comment-hygiene bullets to `AGENTS.md` and byte-identical `.claude/CLAUDE.md`.
- Add `docs/guides/comment-hygiene.md` (taxonomy: keep vs remove; prefer refactor over clarifying comments).
- REMOVE-only scrub of clear dumb comments in `packages/agent`, `packages/core`, `packages/contract`, `packages/db`, `apps/web` per the guide — including numbered pipeline step labels and lettered substeps that only narrate.
- Preserve why/invariant/ADR/ALW comments and functional directives (`biome-ignore` with reason, etc.).

## Acceptance criteria

- [ ] AC-1: `AGENTS.md` + `.claude/CLAUDE.md` include lean comment rules and link the guide (files byte-identical)
- [ ] AC-2: `docs/guides/comment-hygiene.md` defines keep vs remove with examples aligned to owner philosophy
- [ ] AC-3: Clear dumb comments removed from hotspots (banners, name-echo JSDoc, narration, numbered/lettered step labels, JSX section banners); why-comments and directives retained
- [ ] AC-4: No behaviour change — `pnpm typecheck && pnpm lint:check && pnpm test` pass

## Non-goals

- Large refactors to replace comments with named helpers (defer REFACTOR-THEN-REMOVE).
- Editing Markdown product docs beyond the new guide + AGENTS link.
- Platform `sfab` repo cleanup / `code-smells.md` (follow-up).
- Stripping license headers or open TODO/FIXME that reference real work.

## Boundaries

- ✅ Always: prefer self-descriptive code; keep only non-inherent why; preserve directives with reasons; REMOVE-only in this PR.
- ⚠️ Ask first: removing a comment that looks like why but is ambiguous; any non-trivial refactor.
- 🚫 Never: change runtime behaviour; strip `biome-ignore`/`@ts-expect-error` that still apply; edit `.sfab/*`.
