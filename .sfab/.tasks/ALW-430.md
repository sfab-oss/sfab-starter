# ALW-430 Invoice create/edit UX redesign (dogfood B-8) — humane money input, line items, layout

## Context

Dogfood QA (2026-07-08) deferred B-8: the invoice create/edit flow works functionally but the UX is weak on both viewports — the line-item add row is crude, and quantities/prices/tax are entered as raw minor units and basis points (`documents/$id.tsx`), which is developer-grade and error-prone. This is the flagship flow of the ERP base and should model best practice. **Needs a sharpening pass with the user before implementation** — the user said they would spec the ideal flow.

## Scope (provisional, to sharpen)

- Humane money/tax input: currency-formatted fields that convert to minor units / bps under the hood.
- Line-item editing: proper row layout, add/edit/remove affordances, mobile-friendly composition.
- Document detail layout: clear draft vs finalized presentation, payment section hierarchy.
- Entity association beyond "Walk-in" if entity UI lands by then.

## Acceptance criteria (provisional)

- [ ] AC-1: A user can compose an invoice entering prices as normal currency amounts (e.g. 12.50), never raw minor units.
- [ ] AC-2: Line-item add/edit/remove is usable one-handed on a phone viewport.
- [ ] AC-3: Draft vs finalized states are visually distinct and finalized docs read as records, not forms.
- [ ] AC-4: QA walkthrough evidence on desktop + mobile.

## Non-goals

- Settlement-engine changes; new document types (separate decisions from the ALW-426 gap report).

## Boundaries

- ✅ Always: keep API contracts stable unless the sharpening decides otherwise.
- ⚠️ Ask first: schema or API shape changes.
- 🚫 Never: silent rounding changes to money math.
