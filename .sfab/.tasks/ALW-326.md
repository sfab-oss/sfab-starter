# ALW-326 Scaffold the knowledge layer + Tier-1 primitive skeleton (ADR-0010 §E/§H)

## Context

The template architecture is settled in **ADR-0010 (Amendment 2026-06-18)** and **ADR-0017**. This task lays down the *structure* those ADRs imply — the **knowledge layer** (canonical `docs/`, droppable `apps/docs` renderer, `.agents/skills` + runtime symlinks, `AGENTS.md`) and a **skeleton** for the Tier-1 base primitives — **without authoring content**. The repo already has much of this (`apps/docs` exists; `.agents/skills` + the `.claude`/`.agent` symlink farms exist), so the job is to reshape what's there to match the ADRs and add the missing scaffolding. Real docs, real skills, and real primitive implementations are deliberately **separate, later tasks**.

## Scope

- **Canonical docs at repo root.** Establish `docs/` as the single source of truth (`architecture/`, `patterns/`, `decisions/` per ADR-0010 §H). If markdown currently lives in `apps/docs/content/`, move it to root `docs/` and make `apps/docs` a **thin renderer** that reads from it (no duplicated content).
- **Droppable docs app.** Record the fabrication rule: drop the `apps/docs` *app* at fabrication, **keep `docs/`** (a fabrication-ignore marker or documented step).
- **Skills home.** Confirm `.agents/skills/` is canonical and the committed `.claude/skills` + `.agent/skills` symlink farms resolve to it. Add **one placeholder** template-architecture skill (e.g. `data-access`) whose `SKILL.md` is a thin trigger that **points into `docs/` by plain path** (no real content) — to prove the ours-vs-external pattern.
- **AGENTS.md** as the map: where docs live, the skills index, read-before-you-act. Align the existing file.
- **Tier-1 primitive skeleton.** Lay out folders/stubs for the Tier-1 primitives (ADR-0010 §E) where they'll live (schema / core / routes / components) — directories + minimal stubs that typecheck, **not** implementations. Just enough that the parts list is legible in the tree.
- **Manifest shape.** Ensure `.sfab/template.json` can carry the `packs` provenance field (ADR-0017 §3) — type/shape only, no behavior.

## Acceptance criteria

- [ ] AC-1: Canonical `docs/` exists at repo root with the §H subdirs; `apps/docs` renders from it with no duplicated content. verify: `pnpm --filter docs build` renders root `docs/`; grep that content is not duplicated under `apps/docs`.
- [ ] AC-2: Fabrication drops the `apps/docs` app but keeps `docs/`. verify: a fabrication-ignore marker or documented step lists `apps/docs` for removal and `docs/` for retention.
- [ ] AC-3: `.agents/skills/` is canonical; `.claude/skills` + `.agent/skills` are committed symlinks to it; a placeholder template-architecture skill references a doc by plain path. verify: `ls -la .claude/skills` shows symlinks; the placeholder `SKILL.md` contains a `docs/...` path mention, not a symlink.
- [ ] AC-4: `AGENTS.md` maps the knowledge layer (docs location, skills, read-before-act). verify: read `AGENTS.md`.
- [ ] AC-5: Tier-1 primitive skeleton folders/stubs exist and are legible in the tree, matching ADR-0010 §E. verify: directory listing vs the parts list.
- [ ] AC-6: `pnpm typecheck && pnpm lint:check && pnpm test && pnpm build` pass. verify: run all four.

## Non-goals

- Authoring real docs content (code-smells, architecture writeups) — separate content tasks.
- Authoring real skills content — separate.
- Implementing any Tier-1 primitive (Transaction Core etc.) — separate build tasks.
- The registry/pack mechanism (`registry.json`, install skills, manifest *behavior*) — [[ALW-324]] / ADR-0017.
- De-verticalizing the existing example — [[ALW-299]].

## Boundaries

- ✅ Always: structure only; stubs must typecheck; match the ADR-0010 §E/§H + ADR-0017 §6 layout.
- ✅ Always: skills reference docs by path mention; symlinks only for runtime exposure of `.agents/skills`.
- ⚠️ Ask first: deleting/relocating existing docs content beyond the `docs/` move; changing how the template is tagged/released.
- 🚫 Never: author primitive implementations or real doc/skill content here; couple anything to a vertical's domain naming.
