# ALW-531 Document create + line editor + payment wide-dialog forms

## Context

ALW-530 fixed master-data create (wide dialog) + detail-page edit craft for products/entities. Documents and payments still use cramped or prototype forms: create document is a tiny entity-only dialog that navigates away; draft lines are blur-to-save without RHF/zod/headers; record payment is a buried sidebar card with silent invalid no-ops.

Same form grammar as ALW-530: **wide dialog for create / short money capture**, **detail page for heavy edit**, no side sheets.

## Scope

- **Create document:** replace the tiny entity-only dialog with a **wide dialog** that collects the real create header fields already required today (entity + whatever type/direction the API needs); on success navigate to `$id` for lines (keep post-create → detail).
- **Draft document line editor (on `$id`):** real form craft — labeled columns, RHF+zod (or equivalent validated submit), visible errors, no silent blur-save failures; sticky/clear primary actions for add-line / save where applicable. Keep edit on the detail page (not a dialog).
- **Record payment:** move the sidebar stub into a **wide dialog** opened from the document detail (and list row CTA if one already exists); show folio/entity/saldo context in the dialog header; validate amount with visible errors; keep money writes non-optimistic.
- Shared craft: dense muted labels, clear Cancel/Submit, dirty-gate only where it fits (edit-style upserts — **not** on Pay / Finalize).

## Acceptance criteria

- [ ] AC-1: Documents list “New document” opens a wide create dialog (not the cramped entity-only stub); submit creates and lands on `$id`. verify: browser `/documents`
- [ ] AC-2: Draft `$id` line editor has labeled qty/price/tax (or equivalent) columns and shows validation errors instead of silent no-ops. verify: browser draft document
- [ ] AC-3: Finalize / lifecycle CTAs remain on the detail page (not moved into dialogs). verify: draft + finalized detail
- [ ] AC-4: Record payment opens as a wide dialog with document context; invalid amount shows an error; successful pay updates balance. verify: finalized invoice with balanceDue
- [ ] AC-5: No side sheets introduced. verify: code review
- [ ] AC-6: Typecheck/lint/tests green for touched packages. verify: pnpm web checks

## Non-goals

- Side sheets / slide-overs.
- Collect hub (`/collect`), multi-allocate, wallet flows.
- Verb-first nav / Hoy dashboard / settings rewrite.
- Product/entity forms (ALW-530).
- Changing money conservation rules or inventing new document fields.

## Boundaries

- ✅ Always: reuse contract schemas where they exist; AlertDialog for irreversible finalize/void if already used; money never optimistic.
- ⚠️ Ask first: changing which fields belong in create vs only on `$id`; list-row “Pay” CTA if none exists today.
- 🚫 Never: sheets; optimistic payment UI; push to `main`.
