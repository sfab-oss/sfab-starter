# ALW-299 Generalize the inventory vertical into the neutral base + expose pack seams (ADR-0010 §E)

## Context

The template still ships an inventory-vertical worked example (products, warehouses, stock movements, inventory AI tools, `/inventory` routes). [[ALW-47]] originally scoped this as "strip the vertical"; **ADR-0010 (Amendment 2026-06-16 §A)** re-scoped it to **capability-complete + domain-generic** — generalize, don't delete. **Amendment 2026-06-18 §E** now gives the concrete target: a **country-neutral base** shaped around the Tier-1 parts list (Identity, Entity & Contact, Catalog, Transaction Core, File Hub, Activity & Audit Log, …), with country/vertical specialties living in **packs**.

Two jobs follow: **(1) generalize** the existing example to neutral naming aligned with the parts list, and **(2) expose the seams** that capability packs (POS, CFDI, …) graft onto — additively, per **ADR-0017**, so packs extend rather than mutate core.

**The seams + the hub schema are now DESIGNED and MERGED in [[ALW-328]]** — see starter **ADR-006** (`docs/decisions/006-transaction-core.md`) and the full spec **`docs/architecture/transaction-core.md`**. This task **implements that design**; it does not invent or re-decide it. ALW-328 was reached via a 12-tradition + web-research pass, an 8-scenario stress-test, and a 7-agent red-team — so the implementation must honor the **red-team invariants C1–C9** and the **AI-review forward notes** captured below, not just the happy path.

New projects should `pnpm install && dev` and pass `typecheck`/`lint`/`test` with **nothing to delete** before building their own domain.

## Scope

### Keep (capability-complete)
- Auth + org/multi-tenancy; D1 + Drizzle scaffold (migrations, services, Hono RPC); search, uploads, email; the in-app AI assistant with example tools (until [[ALW-218]] replaces the agent); shadcn UI + sidebar + dashboard shell.

### Generalize (domain-generic, parts-list-aligned)
- Replace inventory-vertical naming with **domain-neutral** entities/routes/components/tables that read as the Tier-1 parts (e.g. products/warehouses → generic **Catalog**/location; the order/stock flow → generic **Transaction Core** Documents + LineItems). Exact names TBD in implementation, but must align with ADR-0010 §E and not read as "inventory ERP."
- Re-theme UI copy, nav, charts, AI tools to the neutral example.
- Keep one end-to-end CRUD + dashboard + AI-tool demo showing the full-stack pattern.

### Implement the Transaction Core hub + seams (per ADR-006 / `transaction-core.md`)
Implement the design exactly as `transaction-core.md` specifies. Do **not** define seam shapes here; consume ALW-328's. The load-bearing pieces:
- **The hub:** one `documents` + one `line_items` table, discriminated by `type`/`status`/`direction`, with a **persisted `family` column (CHECK-linked to `type`)** and **no default branch** in `core.documentFamily()` (red-team **C9** — families must be structural, not just a `core` convention). Immutable fiscal successors; transition guards in `contract`/`core`.
- **Money as integer minor units (REVERSAL):** replace the placeholder `numeric → number` money customType with a branded **`MoneyMinor`** integer customType + a single `core/money` value helper (add/sub/allocate/round). **Migrate Catalog `price`/`cost`** and **update the `AGENTS.md` (+ mirrored `.claude/CLAUDE.md`) "money is a numeric → number customType" convention line.** (This reverses [[ALW-327]] D3 / starter ADR-004's money note; recorded in ADR-006.)
- **The money subgraph:** `payments` (signed tender header, open `method`, `idempotencyKey`, `reversesPaymentId`), `payment_allocations` (paymentId↔documentId↔amount, `effectiveAt`, `reversedAt`; **`unique(orgId, paymentId, documentId)`** — red-team **C4**), and the append-only **`customer_credit`** wallet (signed, **nullable `entityId`** for walk-ins — red-team **C3**). Plus `sequences(orgId, key)` numbering bumped at finalize.
- **Projections + invariants:** `amountPaid`/`balanceDue`/`paymentStatus`/`entity.balance`/`creditBalance` are derived, with first-class **full-scan `rebuild*`** (`lastAppliedPaymentId` is a perf hint only) and **reversals as compensating rows** (red-team **C6**). Enforce the **one conservation identity** in `core.recordPayment`/`finalize`: a deposit/overpay lives only in the wallet (never also an AR allocation); `entity.balance` is **sales-fiscal AR only** so supplier `bill`s (AP) never poison fiado (red-team **C1/C2**). Immutability redefined as "lines/totals/tax/identity frozen; payment/posting columns advance" (red-team **C5**).
- **The event spine + critical-tier seam:** two-tier dispatch (`critical` in-tx / `afterCommit`); the base owns folio reservation in the critical tier and exposes it as a **seam packs register into**; the inventory pack registers the **stock decrement** as a critical handler (atomic with finalize, pack-owned), gated by `shouldAffectStock(line)` = catalog `tracksInventory` × line `fulfillmentMode`. The **GL posting hook runs `afterCommit`** (red-team **C7**).
- **AI-review forward notes to resolve during build:** (a) make `payment_allocations` **upsert semantics explicit in `contract`** (re-allocating the same payment→doc pair is an UPDATE under the unique index, not a duplicate INSERT); (b) define the **walk-in wallet matching strategy** (reference/session) for orphaned walk-in credit; (c) define **exactly when `sale_completed` fires** vs `document_finalized`.

### Schema / migration
- Generalize `packages/db/src/schema/inventory.ts` + tables and add the Transaction Core tables (`documents`, `line_items`, `payments`, `payment_allocations`, `customer_credit`, `sequences`) via new Drizzle migrations (follow `defer_foreign_keys` migration conventions). Update services (`packages/core/`), types, routes, hooks, components, tests. The money-customType swap touches every money column — sequence the migration carefully.

- [ ] AC-5: README documents the **base contract** (what ships vs what the AI adds) per ADR-0010 §A/§E and no longer tells users to delete example features to start. verify: read README.
- [ ] AC-6: `pnpm typecheck && pnpm lint:check && pnpm test && pnpm build` pass with zero manual deletions required after clone. verify: run all four.
- [ ] AC-7: Money is **integer minor units** end-to-end — the `MoneyMinor` customType + `core/money` helper replace `numeric → number`; Catalog `price`/`cost` migrated; the `AGENTS.md` + `.claude/CLAUDE.md` money convention line updated. verify: grep for the old numeric-money customType (none in domain tables); read the money helper + the AGENTS.md line.
- [ ] AC-8: The money subgraph exists per `transaction-core.md` §4/§5 — `payments` (header + `idempotencyKey` + `reversesPaymentId`), `payment_allocations` (with `effectiveAt`/`reversedAt` and `unique(orgId, paymentId, documentId)`), `customer_credit` (nullable `entityId`), `sequences`. verify: schema + the unique index.
- [ ] AC-9: The red-team invariants are enforced in `core` (not just documented): C1 one conservation identity (deposit/overpay wallet-only, no double-count); C2 `entity.balance` is sales-fiscal AR only (AP `bill`s excluded); C5 fiscal immutability = totals/lines frozen, payment columns advance; C6 projections rebuild via a full-scan `rebuild*` and reversals are compensating rows. verify: `core` unit tests for each invariant (double-pay, anticipo, AP isolation, reverse-old-payment rebuild).
- [ ] AC-10: `documents.family` is a **persisted column CHECK-linked to `type`** and `core.documentFamily()` has no silent default branch; a stock decrement runs as a **pack-registered critical handler** (atomic with finalize) gated by `shouldAffectStock`. verify: schema CHECK + a test that an unknown `type` throws, and that a finalize+stock decrement commit/rollback together.
- [ ] AC-11: The three AI-review forward notes are resolved: allocation upsert semantics explicit in `contract`; a walk-in wallet matching strategy implemented; the `sale_completed` vs `document_finalized` firing conditions defined and tested. verify: read `contract` allocation schema + the agent/event code + tests.

## Non-goals

- Removing the worked example entirely (forbidden — not a blank slate).
- Designing the Tier-1 primitives / Transaction Core seams — that is [[ALW-328]] (this task consumes its merged output: ADR-006 + `transaction-core.md`).
- The registry / pack mechanism (`registry.json`, install skills, manifest) — [[ALW-324]] / ADR-0017.
- Authoring a base-contract ADR — it already exists (ADR-0010 Amendment 2026-06-18); align to it, don't re-author.
- A contabilidad/GL pack, `line_tax_components` (multi-tax), the CFDI/anticipo netting pack, the financial-family table split, and functional-currency implementation — all **deferred seams** named by ADR-006, not built here.
- `sfab add <recipe>` — dropped per ADR-0010. Think agent port — [[ALW-218]].

## Boundaries

- ✅ Always: keep auth/search/uploads/AI infrastructure; generalize flavor, don't strip capabilities.
- ✅ Always: domain-neutral naming even if the example trends ERP/POS — direction via patterns, not coupled names; seams additive (ADR-0017) and sourced from ALW-328.
- ✅ Always: implement the money subgraph + conservation invariants as `core`-enforced with tests — the schema under-advertises them (red-team finding).
- ⚠️ Ask first: changes to shared packages beyond renaming/generalizing the example; dropping a cross-cutting capability; any deviation from `transaction-core.md`'s shapes.
- 🚫 Never: reintroduce recipe/`add` layers; delete the worked example; change the sfab platform repo; define seam shapes here instead of consuming ALW-328's; ship a base GL or multi-tax (those are deferred packs).
