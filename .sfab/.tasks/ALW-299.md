# ALW-299 Generalize inventory vertical into domain-neutral worked example (ADR-0010 base contract)

## Context

The template still ships an inventory-vertical worked example (products, warehouses, stock movements, inventory-specific AI tools/skills, `/inventory` routes). [[ALW-47]] originally scoped this as "strip the vertical," but **ADR-0010 (amendment 2026-06-16)** re-scoped the base contract to **capability-complete + domain-generic**: keep cross-cutting capabilities (auth, org tenancy, search, uploads, in-app AI assistant) and **generalize** the vertical example into one neutral, re-themeable worked example — not delete it.

New projects should `pnpm install && dev` and pass `typecheck`/`lint`/`test` with **nothing to delete** before building their own domain. The AI re-themes and extends the working example rather than composing onto a blank shell.

## Scope

### Keep (capability-complete)
- Auth + org/multi-tenancy (Better Auth, onboarding, invitations).
- D1 + Drizzle scaffold (migrations, services pattern, Hono RPC).
- Search, uploads, email infrastructure.
- In-app AI assistant with example tools wired to the worked example (until [[ALW-218]] replaces the agent architecture).
- shadcn UI + sidebar layout + dashboard shell.

### Generalize (domain-generic)
- Replace inventory-vertical naming with **domain-neutral** entities, routes, components, tables, and types (e.g. products/warehouses → generic catalog/location concepts — exact names TBD in implementation, but must not read as "inventory ERP").
- Re-theme UI copy, nav labels, charts, and AI skills/tools to match the neutral example.
- Keep one end-to-end CRUD + dashboard + AI-tool demo that shows the full-stack pattern.
- Update README: remove "delete inventory to start" instructions; document the **base contract** (what ships vs what the AI adds per project) per ADR-0010 §A.
- Add a short ADR or doc section in `docs/decisions/` capturing the capability-complete + domain-generic contract.

### Schema / migration
- Rename/generalize `packages/db-d1/src/schema/inventory.ts` and related tables via a new Drizzle migration (follow existing migration conventions in `docs/decisions/003-d1-migrations-monorepo.md`).
- Update services (`packages/core/`), types (`packages/types/`), API routes, hooks, components, and tests accordingly.

## Acceptance criteria

- [ ] AC-1: No inventory-vertical-specific naming remains in user-facing routes, nav, or table names (grep for `inventory`, `warehouse`, `product` as domain terms — infra package names like `web-tanstack` are fine). verify: targeted grep + manual nav review.
- [ ] AC-2: One neutral worked example still demonstrates full-stack CRUD + dashboard + AI tool calling end-to-end. verify: `pnpm dev`, exercise the example feature and chat tools.
- [ ] AC-3: README no longer instructs users to delete example features to start building; includes a "base contract" section aligned with ADR-0010. verify: read README.
- [ ] AC-4: `pnpm typecheck`, `pnpm lint:check`, `pnpm test`, and `pnpm build` pass with zero manual deletions required after clone. verify: run all four on a fresh mindset (no leftover inventory files).
- [ ] AC-5: Base contract documented in `docs/decisions/` (new ADR or amendment note referencing ADR-0010). verify: file exists and lists kept capabilities vs AI-added domain.

## Non-goals

- Removing the worked example entirely (ADR-0010 amendment forbids "business-empty blank slate").
- `registry.json` / shadcn GitHub registry setup (deferred from v1 per ADR-0010 amendment).
- `sfab add <recipe>` mechanism — dropped per ADR-0010.
- Think/DO agent port — [[ALW-218]] (separate task; may follow or parallelize after naming stabilizes).
- Porting ALW-47's platform-side planning text — this task is the **implementation** on the template repo.

## Boundaries

- ✅ Always: keep auth, search, uploads, and AI assistant infrastructure; generalize domain flavor, don't strip capabilities.
- ✅ Always: domain-neutral naming even if the example later trends ERP/POS — direction via patterns, not coupled names.
- ⚠️ Ask first: changes to shared packages beyond renaming/generalizing the example (`@workspace/types`, `@workspace/core`).
- ⚠️ Ask first: dropping a cross-cutting capability from the base.
- 🚫 Never: reintroduce recipe/`add` install layers; delete the worked example; change the sfab platform repo.
