# ALW-394 Triage and clear Dependabot alerts

## Context

Pushing to `main` surfaces a GitHub Dependabot banner (headline figure ~132 vulnerabilities). The authoritative Dependabot alerts API tells a smaller, more actionable story — as of 2026-07-04 there are **38 open alerts** (the banner over-counts; anchor triage on the API):

| Severity | Open |
|---|---|
| Critical | 0 |
| High | 7 |
| Moderate | 28 |
| Low | 3 |

The 7 **high** alerts, all with a patched version available:

| Package | Vulnerable range | First patched | Likely origin |
|---|---|---|---|
| `undici` | `>= 7.0.0, < 7.24.0` (x3) | 7.24.0 | transitive (wrangler/miniflare tooling) |
| `undici` | `>= 7.0.0, < 7.28.0` | 7.28.0 | transitive |
| `ws` | `>= 8.0.0, < 8.21.0` | 8.21.0 | transitive |
| `hono` | `< 4.12.25` | 4.12.25 | **direct** (API layer) |
| `lodash` | `>= 4.0.0, <= 4.17.23` | 4.18.0 | transitive |

Most are transitive and clear by refreshing `pnpm-lock.yaml`; `hono` is a direct dependency and needs a version bump. (13 further alerts are already auto-dismissed and need no action.)

## Scope

- Pull the current open alert set from the Dependabot API and confirm counts (they move as upstream publishes advisories).
- Bump `hono` to `>= 4.12.25` (direct dep) and update the lockfile.
- Refresh transitive deps (`undici`, `ws`, `lodash`, and the 28 moderates) via `pnpm update` / lockfile regeneration; use `pnpm.overrides` only where a transitive can't be pulled forward by its parent.
- For any alert with no safe fix (breaking major, or a dev/tooling-only path with no exploit surface in this app), dismiss it in GitHub with a written reason rather than leaving it open.
- Verify the app still builds and passes the full gate after the bumps.

## Acceptance criteria

- [ ] AC-1: Every open **high** and **critical** Dependabot alert is either patched (dependency updated) or explicitly dismissed with a documented reason.   verify: `gh api '/repos/sfab-oss/sfab-starter/dependabot/alerts?state=open' -q '.[].security_advisory.severity' | sort | uniq -c` shows no `high`/`critical`
- [ ] AC-2: `hono` in `apps/web` resolves to `>= 4.12.25`.   verify: `pnpm --filter web why hono` / lockfile
- [ ] AC-3: Full gate green after the updates.   verify: `pnpm typecheck && pnpm lint:check && pnpm test && pnpm build`
- [ ] AC-4: Any alert left open is intentional and annotated — a short note in the PR lists each remaining moderate/low with why it's deferred (no safe fix / not reachable).

## Non-goals

- Chasing every moderate/low to zero if a fix is a breaking major with no real exploit path in this app — document and defer instead.
- Changing runtime behavior or app features; this is a dependency-hygiene pass only.
- Adopting new tooling (Renovate, etc.) — that's a separate decision.

## Boundaries

- ✅ Always: keep bumps within the same major where possible; run the full gate before merge; prefer fixing at the direct-dep or override level over patching lockfile entries by hand.
- ⚠️ Ask first: any **major**-version bump of a direct dependency (`hono`, `better-auth`, `drizzle-orm`, `wrangler`, TanStack/Vite) — these can carry breaking changes.
- 🚫 Never: dismiss a high/critical alert without a written justification; hand-edit `pnpm-lock.yaml`; introduce a `pnpm.overrides` pin without a comment explaining why.
