# ALW-424 QA: auth, onboarding & org lifecycle — incl. RBAC gating

## Context

The account lifecycle (signup → onboarding → org create → invite → accept → roles) has never had a dedicated QA pass. Known landmine from the surface map: the login page links to `/forgot-password` but no such route exists (dead link). RBAC spine ([[ALW-347]]: owner/admin/operator) gates settings actions and needs behavioral verification per role, not just UI-disabled states.

## Scope

- Signup (validation, duplicate email), login (bad credentials, toast quality), logout, session-expiry behavior mid-session.
- Onboarding: org creation, live slug derivation, slug-collision handling.
- Invitations: invite from settings, email/role constraints, accept and decline via `accept-invitation.$id`, error card for invalid/expired ids, pending-invitations table.
- RBAC: for each role (owner/admin/operator) verify what settings/member actions are visible, disabled, and — critically — rejected server-side if attempted.
- Multi-org: second org creation, membership isolation (data from org A never visible in org B).

## Acceptance criteria

- [ ] AC-1: Full lifecycle walked with evidence: signup → onboarding → invite (each role) → accept → role-gated actions.
- [ ] AC-2: Dead `/forgot-password` link confirmed and filed (fix or remove; note there is no password-reset flow at all).
- [ ] AC-3: RBAC matrix documented (role × action → allowed/denied) with at least one server-side rejection probe per restricted action.
- [ ] AC-4: Org data isolation spot-checked across two orgs (catalog + documents + chat history).
- [ ] AC-5: Findings logged and filed as draft tasks.

## Non-goals

- Building password reset (would be its own task if we decide we need it).
- Agent-tool authorization ([[ALW-400]] covers that design).

## Boundaries

- ✅ Always: throwaway local accounts/orgs; evidence per finding.
- ⚠️ Ask first: anything touching real email delivery.
- 🚫 Never: prod auth config or remote resources.
