# ALW-523 Pre-launch: remove deprecated APIs (no compat shims)

## Context

The starter has not publicly launched yet. We do not need deprecation windows or compat shims — if something is marked `@deprecated` or is an empty stub left for "old callers," delete it and update call sites/tests/docs. Keeping deprecated surfaces adds noise and implies a stability contract we have not promised.

Known starter-owned example from the agent-tools review: `getOrgAgentApprovalTools` is an empty deprecated stub (approval lives on in-codemode `needsApproval`); still exported and asserted in tests.

## Scope

- Inventory starter-owned `@deprecated` markers and deprecated exports (packages/agent, apps/web, packages/*, docs) — **exclude** generated Cloudflare/worker-configuration noise and third-party type stubs.
- Remove deprecated functions/exports/stubs; update imports, tests, and docs to the current API.
- Prefer delete-and-fix over renaming with a deprecated alias.
- Do not introduce new `@deprecated` shims for pre-launch cleanup.

## Acceptance criteria

- [ ] AC-1: Inventory of starter-owned deprecated items (task comment or packet artifact), with generated/vendor noise explicitly out of scope
- [ ] AC-2: Each inventoried item is removed or rewritten to the current API (no empty deprecated stubs left in public package exports)
- [ ] AC-3: Tests and docs that referenced the deprecated surface are updated; typecheck/tests green

## Non-goals

- Cleaning `@deprecated` inside generated `worker-configuration.d.ts` / Wrangler types.
- Broader agent ToolResult / fail-contract redesign (separate work).
- Public semver / changelog deprecation policy for post-launch.

## Boundaries

- ✅ Always: delete unused deprecated starter APIs rather than keep shims pre-launch.
- ⚠️ Ask first: removing something still referenced by an external template consumer we care about (unlikely pre-launch).
- 🚫 Never: leave empty `@deprecated` stubs "for compatibility" without a real caller.
