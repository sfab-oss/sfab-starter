# ALW-327 Foundational architecture: repo/package structure + Transaction Core and the primitives around it (Tier-1)

## Context

Tier-1's **foundational architecture** task — the design the rest of the build hangs off (ADR-0010 §E). Before implementing primitives, decide two cross-cutting things everything depends on:

- **(A) the monorepo directory/package structure** — how the Tier-1 primitives map onto `packages/` and `apps/`, where schema / core / types / UI / routes live, and where the web, `apps/docs`, and `apps/registry` apps sit.
- **(B) the Transaction Core architecture** — the **hub** (Documents + LineItems, money/tax/folios, status lifecycle, the domain-event spine it emits) and how the surrounding primitives (Catalog, Entity & Contact, Activity & Audit Log, File Hub) relate to it.

This is **design + decision**; the output is documented architecture the build tasks follow. **Implementing** Transaction Core (migrations, services, UI) is a *separate* Tier-1 build task. Transaction Core is the hub the whole Mexican pack-set (POS, CFDI, fiado, …) grafts onto, and POS is its forcing function — so it earns careful design (ADR-0010 §E; ADR-0017).

## Scope

### Part A — Directory / package structure
- Map every Tier-1 primitive (Identity, i18n, Entity & Contact, Catalog, Transaction Core, File Hub, Activity & Audit Log, Reporting, AI Agent, jobs, Notifications) to its home: `packages/` vs `apps/`, and within that where schema / services-core / types / UI / routes live.
- Place the special apps: the web app, `apps/docs` (renderer), `apps/registry` (showcase + build, ADR-0017 §6).
- Naming + package-boundary conventions (domain-neutral; shared-package vs app-local; how `@workspace/*` aliases are organized).
- **Output:** a `docs/architecture/` structure doc (and/or an ADR) that the scaffold ([[ALW-326]]) implements.

### Part B — Transaction Core (the hub) + surrounding primitives
- The Documents + LineItems model (quote = order = invoice = PO, one structure); decimal-safe money; the generic tax field; document numbering/folios; the status lifecycle.
- The **domain-event spine** it emits (`sale_completed`, …) — the shared backbone feeding pack integration (the hook), the event log (BI), and the activity timeline (ADR-0010 §F).
- How the surrounding primitives relate: Catalog (line items reference products), Entity & Contact (counterparty + running balances), Activity & Audit Log (events + history), File Hub (attached documents).
- The **seams** it exposes for packs (ADR-0017; cf. [[ALW-299]]): optional 1:1 compliance-doc relation, open `paymentMethod`, etc.
- **Output:** a `docs/architecture/transaction-core.md` (and/or an ADR) — design, not full implementation.

## Acceptance criteria

- [ ] AC-1: A directory/package-structure doc maps every Tier-1 primitive to its home (packages/apps; schema/core/UI/routes) plus the web/docs/registry apps. verify: read the structure doc.
- [ ] AC-2: Naming + package-boundary conventions are documented (domain-neutral; shared-vs-app-local; `@workspace/*` layout). verify: read the doc.
- [ ] AC-3: A Transaction Core architecture doc defines the Documents+LineItems model, money/tax/folios, status lifecycle, and the domain-event spine. verify: read `docs/architecture/transaction-core.md`.
- [ ] AC-4: The doc shows how Catalog, Entity, Activity & Audit Log, and File Hub relate to Transaction Core, and which seams it exposes for packs. verify: read the doc + cross-check ADR-0017 / [[ALW-299]] seams.
- [ ] AC-5: The structure decision is concrete enough that [[ALW-326]] can lay out the skeleton from it. verify: ALW-326's Tier-1 folders map 1:1 to this doc.

## Non-goals

- **Implementing** Transaction Core (schema migrations, services, UI) — a separate Tier-1 build task.
- The other primitives' detailed internal design beyond how they relate to the hub.
- The registry / pack mechanism — [[ALW-324]] / ADR-0017.
- Generalizing the existing inventory example — [[ALW-299]] (this informs its target naming).

## Boundaries

- ✅ Always: domain-neutral naming; design to the ADR-0010 §E parts list + §F event spine; keep it *legible, not optimal* (ADR-0010 §D stopping line).
- ⚠️ Ask first: introducing a new top-level package/app convention that diverges from the current monorepo.
- 🚫 Never: couple the structure or schema to a vertical/country; over-engineer a plugin framework (generosity over seams, ADR-0010 §D).
