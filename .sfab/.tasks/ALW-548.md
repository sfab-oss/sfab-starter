# ALW-548 Rich chat input with typed mention/command triggers (InputGroup-native)

## Context

The starter's chat inputs are plain textareas composed on shadcn `InputGroup`. They cannot render inline mention chips or offer trigger-based typeahead (`@` for members/agents, `/` for commands) — the primitive the chat dock needs for hybrid slash/chat interactions (see [[ALW-533]]). TipTap v3 dependencies are already staged in `packages/ui/package.json` but unused. This task introduces a reusable rich chat-input component and proves it in the chat registry blocks.

## Scope

- New component `packages/ui/src/components/ai-elements/chat-input.tsx` built on TipTap, composing shadcn `InputGroup` idioms (the editor acts as the group's control slot).
- Component-first typed API: a generic `mentions` prop (`{ [key]: { trigger, items, render? } }`) whose item types flow into `onSubmit(parsed)` — `parsed.text: string` plus one typed array per mention key. No required hook, no double declaration of configs.
- Submit semantics: Enter submits, Shift+Enter inserts a newline; `onSubmit(parsed, { clear, focus })` — the caller decides when to clear (no auto-clear); `status`-driven submit button (send / loading / stop).
- Suggestion popup anchored to the caret, keyboard-navigable (up/down/enter/escape), with a default item renderer and per-config `render` override; items may be a static array or an async `(query) => items` function.
- Wire the component into both chat registry blocks (`packages/registry/registry/blocks/chat-page`, `.../chat-dock`), replacing their textarea composers: `@` member mentions in both, `/` commands in the chat dock, backed by the blocks' existing mock data.
- Sizing contract: auto-grow with a default max height on the editor, overridable per instance via `className`; no dedicated sizing props.

## Acceptance criteria

- [ ] AC-1: `packages/ui` exports `ChatInput`, `ChatInputEditor`, `ChatInputSubmitButton`, `ChatInputMentionButton` from `ai-elements/chat-input.tsx`; consumers compose layout with existing `InputGroupAddon`/`InputGroupButton` (no pass-through layout wrappers are shipped).
- [ ] AC-2: With `mentions` declared once as a prop, `onSubmit`'s `parsed` is statically typed per mention key (verify: hover/type-test in a registry block — `parsed.member` is the member item array type, `parsed.text` is string).
- [ ] AC-3: The editor emits `data-slot="input-group-control"` and inherits InputGroup focus/invalid styling; it auto-grows to a default max height and scrolls beyond it while block-end addons stay pinned.
- [ ] AC-4: Enter submits, Shift+Enter adds a newline; typing a trigger character opens the suggestion popup filtered by the query; arrow keys + Enter select; Escape closes; selected mentions render as inline chips.
- [ ] AC-5: `parsed.text` is plain text (mentions appear as trigger+label, no HTML markup); each mention key yields a deduplicated array of the selected items.
- [ ] AC-6: Both chat blocks (`chat-page`, `chat-dock`) use the new component; chat-dock additionally registers a `/` command trigger from mock data; block demo pages render and submit correctly.
- [ ] AC-7: Repo gates pass: `pnpm typecheck`, `pnpm lint:check`, `pnpm test`, `pnpm build` (verify: command output).

## Non-goals

- Migrating `apps/web/src/components/chat/parts/chat-input.tsx` (the product chat input) — follow-up after the registry blocks validate the API.
- Attachments/file-drop handling inside the component (stays in consumer composers).
- Persisted drafts, collaborative editing, or markdown rendering in the editor.

## Boundaries

- ✅ Always: keep the component self-contained in `packages/ui`; use existing shadcn components for popup/menu primitives; follow repo lint/format conventions.
- ⚠️ Ask first: adding any new dependency beyond a positioning utility (e.g. floating-ui) if needed for the suggestion popup.
- 🚫 Never: modify `.sfab/**`, push to `main`, change `input-group.tsx` semantics, or alter the product app's chat input.
