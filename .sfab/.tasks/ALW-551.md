# ALW-551 Experimental proposal: org chat @-mention entity grounding via rich chat input

## Context

**Status: experimental proposal** — filed to evaluate, not yet committed direction.

The org-agent chat solves record grounding only for the page the user is currently on: `packages/agent/src/context/page-context.ts` injects the current page's entity type/id into the prompt ("IDs are the tool handle") so the agent fetches exact state via `get_*` tools. Any record *not* on the current page must be found by fuzzy name search inside the conversation ("compare Acme's balance with Initech's"), which is slower and error-prone.

[[ALW-548]] shipped a reusable rich chat input (`packages/ui/src/components/ai-elements/chat-input.tsx`) with typed `@` mention triggers, proven in the chat registry blocks. An `@`-mention of an entity generalizes the page-context mechanism: the user picks the exact record; the agent receives `{type, id, name}` deterministically.

## Scope

- Swap the internals of `apps/web/src/components/chat/parts/chat-input.tsx` to the ui `ChatInput`/`ChatInputEditor`/`ChatInputSubmitButton`, deleting the local textarea composer, hand-rolled submit button, and Enter-key handling. App-specific concerns (page-context chip, image capability gating, error toasts, voice controller) stay consumer code composed in the addons.
- One mention type, `entity`, with an async `items: (query) => …` backed by the existing entity list/search query hooks.
- Contract: optional `mentions` array (`{type, id, name}`) added to `aiMetadataSchema.pageContext`'s sibling position in `packages/contract/src/ai.ts` (outgoing user-message metadata).
- Agent: a `buildMentionsSection` sibling to `buildPageContextSection` — "Referenced records: …, fetch current state via the matching get_* tool with these ids."

## Acceptance criteria

- [ ] AC-1: Typing `@` in the org chat opens a typeahead over org entities (async search); selecting renders an inline chip.
- [ ] AC-2: Submitted messages carry `metadata.mentions` with the selected records' `{type, id, name}`; `parsed.text` stays plain text.
- [ ] AC-3: The agent prompt includes a referenced-records section when mentions are present, and answers about a mentioned record use `get_*` tool calls keyed by the mentioned id (verify: transcript of a chat mentioning an off-page entity).
- [ ] AC-4: The app's local `ChatSubmitButton` copy is deleted; stop/loading/error states behave as before (stop only offered when `onStop` exists).
- [ ] AC-5: Voice input still works via the `ChatInputHandle` (`getText`/`setText`/`clear`) with no shadow text state in the composer.
- [ ] AC-6: Bundle impact measured and reported: TipTap's addition to the main app chunk, and whether the chat dock/panel is (or should be) lazy-loaded to contain it. verify: build output comparison before/after.
- [ ] AC-7: Repo gates pass: `pnpm typecheck`, `pnpm lint:check`, `pnpm test`, `pnpm build`.

## Non-goals

- `/` slash commands in the product chat — no real chat-level command inventory yet; the API supports adding a second trigger later without redesign.
- Mentioning documents, transactions, or team members — start with entities; widen only if the experiment validates.
- Any change to the registry blocks or the shared ui component API (frozen by [[ALW-548]]).

## Boundaries

- ✅ Always: keep row/input typing per the two-schema-sources rule; mentions metadata is hand-written Zod in `contract`.
- ⚠️ Ask first: if the bundle-size measurement (AC-6) argues for restructuring how the chat dock loads; any new dependency.
- 🚫 Never: modify `.sfab/**`; push to `main`; let `packages/ui` import app/core code.
