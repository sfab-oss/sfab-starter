# ALW-429 Command-K palette: wire real global search (pages + catalog + documents)

## Context

The ⌘K palette (`components/search/search-command.tsx`) looks finished but is a stub: it filters a hardcoded 3-page list (Home, Catalog, Settings — Documents is missing), carries dead scaffolding (`isLoading` hardcoded `false`, a 300ms debounce feeding nothing async), and ignores the existing catalog search endpoint (`hono/org-protected/catalog/search.ts`). For an AI-native ERP base the palette should be the fast path to any record.

## Scope

- Complete the static page list (add Documents; keep it derived from the nav definition rather than duplicated).
- Wire live entity search: products via the existing catalog search endpoint; documents (folio/entity name) via a search-capable endpoint (add one if the documents API lacks it).
- Grouped results (Pages / Products / Documents) with icons, loading state, empty state, and Enter-to-navigate to the right detail route.
- Keep keyboard behavior (⌘K, Ctrl-K, `/`, input-focus guard) intact; keep response snappy (debounce + request cancellation).

## Acceptance criteria

- [ ] AC-1: Typing a product name surfaces catalog results that navigate to `/catalog/$id`.
- [ ] AC-2: Typing a document folio or entity name surfaces document results that navigate to `/documents/$id`.
- [ ] AC-3: All app pages, including Documents, appear as page results.
- [ ] AC-4: Loading, empty, and no-results states render correctly; the `isLoading` scaffold is either real or removed.
- [ ] AC-5: Keyboard-only use works end-to-end (open, navigate results, select, close).

## Non-goals

- Command/action execution from the palette (create product, etc.) — possible follow-up.
- Full-text search infrastructure; simple prefix/LIKE search is fine for the base.

## Boundaries

- ✅ Always: org-scoped queries only; reuse existing hono route patterns.
- ⚠️ Ask first: new DB indexes or schema changes for search.
- 🚫 Never: cross-org leakage in results.
