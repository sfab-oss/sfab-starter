# ALW-434 Customer-credit wallet UI — balance, ledger, deposit/redeem on the entity page

## Context

The credit wallet ([[ALW-355]]) shipped API-only: `wallet.ts` exposes `GET /entries` (by entity), `GET /balance/:entityId`, and `POST /deposit`, `/redeem`, `/redeem-by-reference`. There is no way to see or move customer credit from the UI. The natural home is the entity detail page, which the entities task creates.

## Scope

- Wallet card on entity detail: current balance (major-unit formatted via `@workspace/ui/lib/money`) + ledger of entries (deposits, redemptions, references to documents/payments).
- Deposit flow: dialog with humane currency input, reason/reference; role-gated appropriately.
- Redeem flow: apply available credit against an open document balance (redeem-by-reference), surfaced both from the entity wallet card and as a "pay from credit" option in the document Record Payment area when the entity has balance.
- Empty/zero-balance states that teach the concept (this is a template others learn from).

## Acceptance criteria

- [ ] AC-1: Entity detail shows live balance and ledger for a seeded entity with credit history.
- [ ] AC-2: Deposit through the UI updates balance and ledger immediately.
- [ ] AC-3: Redeeming against an open invoice reduces both the invoice balance due and the wallet balance, consistent with the API's conservation rules.
- [ ] AC-4: Document payment section offers wallet credit only when the linked entity has available balance.
- [ ] AC-5: Browser walkthrough evidence, desktop + mobile.

## Non-goals

- New wallet semantics (expiry, transfers between entities); backend changes beyond what the UI strictly needs.

## Boundaries

- ✅ Always: money displayed in major units, stored/sent in minor units; org-scoped.
- ⚠️ Ask first: any change to wallet API contracts.
- 🚫 Never: client-side money arithmetic that bypasses core conservation.
