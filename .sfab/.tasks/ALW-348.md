# ALW-348 Codemode tool approvals: adopt the supported durable-approval pattern (revisit needsApproval)

## Context

The agent exposes ERP tools to the model through **codemode** — `createExecuteTool({ tools, loader })` in `packages/agent/src/org/chat/org-chat.ts:272`, on `@cloudflare/think@0.8.5` + `@cloudflare/codemode@0.3.8`.

While building RBAC ([[ALW-347]]) we discovered a latent trap: in our installed version, `createExecuteTool` → legacy `createCodeTool` → `filterTools()`, which **silently strips out any tool whose `needsApproval` is `true` or a function** (verified in `@cloudflare/codemode/dist/resolve.js` `filterTools`, used by `createCodeTool` in `dist/ai.js`). So an approval-gated tool never appears in the sandbox at all — the model can't call it, and no error is surfaced. Our catalog write tools all carried `needsApproval: true`, which meant they were invisible to the agent. ALW-347 **dropped `needsApproval`** from those tools and relies on an in-`execute` RBAC `assertCan()` throw instead (autonomous, RBAC-gated — no human approval). That unblocked the agent but means we currently have **no human-in-the-loop approval** on any agent write.

Cloudflare's official guidance (cloudflare/agents repo, `docs/codemode/approvals.md`, `docs/codemode.md:425-429`, `packages/codemode/README.md`, as of upstream 2026-06-17) is **version-dependent**:
- **Legacy `createCodeTool` (our current 0.3.8):** "Tool approval (`needsApproval`) is not supported yet … excluded from codemode … use approval-required tools through standard AI SDK tool calling instead." → keep approval tools **top-level**, out of codemode.
- **Newer durable runtime / connector path:** `createCodemodeRuntime` + connectors mark sensitive tools `requiresApproval: true`; a gated call **durably pauses** the sandbox run and resumes via `runtime.approve({ executionId })` (Think exposes `pendingExecutions()` / `approveExecution()`). `ToolSetConnector` maps AI SDK `needsApproval` → `requiresApproval` instead of stripping it. **This path does not exist in our pinned 0.3.8 dist** (no `createCodemodeRuntime` / `ToolSetConnector` / `requiresApproval` — confirmed).

Note: Cloudflare's own docs are **internally inconsistent right now** (`codemode.md` still says approvals are "planned" while `codemode/index.md` + `approvals.md` document them as shipped) — the feature is mid-rollout, so anything here must be verified against the exact version we land on.

This task is to determine and implement the **correct, supported** way to do agent-tool approvals for this codebase — so that when a write genuinely warrants human confirmation (e.g. `delete-product`, or future money/document actions), we have a real, non-silent mechanism.

## Scope

- Audit the current codemode wiring and confirm the exact stripping behaviour on the installed version (`org-chat.ts` getTools, `createExecuteTool`, `filterTools`).
- Evaluate **upgrading** `@cloudflare/think` + `@cloudflare/codemode` to a version that ships the runtime/connector approval path (`createCodemodeRuntime` / `ToolSetConnector` / `requiresApproval` / `runtime.approve`). Assess breaking changes vs current `createExecuteTool` usage.
- Decide, per tool class, the approval model: **durable in-codemode approval** (`requiresApproval`, pause/approve/resume) vs **top-level standard AI-SDK approval tool** (out of codemode) vs **autonomous + RBAC `assertCan`** (no approval). Document the rule.
- Implement the chosen pattern and wire the **pending / approve / reject** surface into the agent chat UI (approve/reject affordances driven by `pendingExecutions()` / runtime pending, not the raw transcript).
- Add a test proving an approval-gated call actually **pauses and then resumes** (or is correctly held top-level) — closing the gap that hid the original silent-strip bug.

## Non-goals

- Re-deciding RBAC itself — `assertCan` / role-rank from [[ALW-347]] stays; approvals are orthogonal to authorization.
- Building the Transaction Core money/document surfaces — this task provides the approval mechanism they may consume, not the screens.
- Changing the platform repo — tracked separately in the companion platform task.

## Boundaries

- ✅ Always: verify approval behaviour against the exact installed version's source, not docs alone (docs are mid-rollout and inconsistent).
- ✅ Always: keep `assertCan` RBAC enforcement regardless of the approval mechanism chosen.
- ⚠️ Ask first: upgrading `@cloudflare/think` / `@cloudflare/codemode` major/minor (it changes the agent execution path); adding a human-approval requirement to a tool that is currently autonomous.
- 🚫 Never: reintroduce `needsApproval: true` on a codemode tool without confirming the installed version pauses (rather than silently strips) it.
