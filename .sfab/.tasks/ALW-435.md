# ALW-435 Documents UI: expose types & directions beyond the sales invoice (quotes, orders, bills, credit notes)

## Context

The contract supports 10 document types (`quote`, `sales_order`, `purchase_order`, `invoice`, `credit_note`, `receipt`, `bill`, `goods_receipt`, `adjustment`, `transfer`) across `sales`/`purchase` directions, and the documents API already accepts them â€” but the hub's only entry point is "New Invoice" (`type: invoice`, `direction: sales`). The base under-demonstrates its own core. Decide the UI-worthy subset deliberately rather than mechanically exposing all ten.

## Scope

- Document creation: replace the single "New Invoice" button with a typed create flow (suggested first wave: quote, sales order, invoice, credit note on the sales side; purchase order and bill on the purchase side; leave goods_receipt/adjustment/transfer for inventory flows unless trivially free).
- Hub: type + direction filters and clear type badges; sensible grouping or tabs for sales vs purchase.
- Detail page: type-aware presentation (a quote shouldn't show "Record payment"; a credit note shows its disposition â€” cash refund vs wallet credit, tying into the wallet task).
- Type-appropriate lifecycle actions honoring what core already supports (e.g. finalize semantics per type); document any type the core supports but the UI intentionally leaves out, and why.

## Acceptance criteria

- [ ] AC-1: A user can create at least quote, sales order, invoice, credit note (sales) and purchase order, bill (purchase) from the hub.
- [ ] AC-2: Hub filters by type and direction; each row's type is visually distinct.
- [ ] AC-3: Detail renders type-appropriately â€” payment UI only where paying makes sense; credit-note disposition visible.
- [ ] AC-4: Types intentionally not exposed are listed in the PR description with rationale.
- [ ] AC-5: Browser walkthrough evidence, desktop + mobile.
