# ALW-435 Documents experience v1 ‚Äî composer redesign + type system (quote / invoice / credit note / bill)

## Context

Sharpened 2026-07-08 (session on [[ALW-430]]+[[ALW-435]]; ALW-430 is folded in here and canceled). The documents UI exposes a fraction of Transaction Core: hardcoded sales-invoice creation, add-only line items in raw minor units / bps, one payment form. `docs/architecture/transaction-core.md` additionally describes lifecycle mechanics (progression by immutable successor, credit-note correction) that **core does not implement yet** ‚Äî no `createSuccessor`, no line update/remove, no void. This task builds the agreed v1 slice of both core and UI in one deliverable (user's explicit call: one combined PR rather than sequenced tasks).

**Agreed decisions (sharpening session):**
1. Scope depth: UI + core essentials ‚Äî line-item update/remove and successor creation (quote‚Üíinvoice conversion, credit-note-from-invoice). **Deferred:** void/reversal, partial fan-out, `sales_order`/`purchase_order` intermediates, `receipt`, stock family.
2. Type set v1: `quote`, `invoice`, `credit_note` (sales) + `bill` (purchase).
3. Create flow: one "New document" menu in the hub header ‚Äî Quote / Invoice / Bill. Credit notes are **never** created standalone: only via "Create credit note" on a finalized invoice, choosing disposition (refund / store credit / apply) there.
4. Hub: full-width ResourceTable (pattern from [[ALW-421]]/[[ALW-433]]) ‚Äî columns folio, type badge, entity, status, payment status, total, date; type/direction/status filters; URL-synced search. Activity feed leaves the hub (stays on detail).
5. Composer: one `/documents/$id` route, two renderings. Draft = editable line table (product picker snapshotting price/description, description, qty, unit price in **major units**, tax as **percent**; inline edit + remove per row) with entity picker and Finalize in the header. Finalized = same layout frozen into a record.
6. Money entry: major units / tax % everywhere at the UI boundary (ADR-006); minor units and bps never appear in the UI.
7. Payment form: amount pre-fills balance due and is **capped at balance due** in v1 (overpay‚Üíwallet routing deferred until [[ALW-434]] lands); "pay from credit" option arrives with [[ALW-434]], not here.

## Scope

**Core + API (the essentials only):**
- `updateLineItem` / `removeLineItem` in `packages/core/src/transaction/documents.ts` + contract inputs + hono endpoints (draft-only, guarded).
- `createSuccessor(sourceDocId, {type})` implementing the architecture's successor semantics: new header + copied line snapshots, `sourceDocumentId`/`rootDocumentId` lineage; used by quote‚Üíinvoice ("Convert to invoice" on a finalized/accepted quote) and invoice‚Üícredit_note ("Create credit note" + disposition via existing `applyCreditNoteDisposition`).
- `createDocument` accepts the v1 type set from the UI (validation already permits it).

**UI:**
- Hub ‚Üí ResourceTable + filters + "New document" menu (Quote / Invoice / Bill).
- Detail/composer per decisions 5‚Äì7, type-aware: no payment section on quotes; credit note shows disposition and links its source invoice; bill behaves like invoice with direction=purchase (AP side, no wallet); lineage strip ("Converted from Quote #‚Ä¶").
- Type badges consistent across hub, detail, and (if present) command-K results.

## Acceptance criteria

- [ ] AC-1: From the hub I can create a quote, an invoice, and a bill via the New-document menu; credit notes are absent from the menu.
- [ ] AC-2: In a draft, I can add, edit, and remove line items with prices entered in major units and tax in percent; totals update live and match core's computation exactly (verify: API cross-check).
- [ ] AC-3: A finalized quote offers "Convert to invoice"; the result is a new linked invoice draft with copied lines, and the quote shows its lineage.
- [ ] AC-4: A finalized invoice offers "Create credit note" with disposition choice; the credit note links back and the disposition applies via core.
- [ ] AC-5: The payment form pre-fills balance due, rejects amounts above it, and money is entered in major units.
- [ ] AC-6: Hub table filters by type, direction, and status with URL-synced search; each type is visually distinct.
- [ ] AC-7: Quotes show no payment UI; a finalized document's lines/totals are visibly immutable.
- [ ] AC-8: Browser walkthrough evidence, desktop + mobile.

## Non-goals

- Void/reversal UI, partial conversion fan-out, `sales_order`/`purchase_order`/`receipt`/stock types (follow-up tasks when needed).
- Overpayment‚Üíwallet routing and "pay from credit" ([[ALW-434]]).
- Standalone payments view / payment reversal UI.
- Multi-document payment allocation UI.
