# ALW-398 Org-agent (Think) modernization sweep — upgrade, multi-session, filesystem, tools & agent UX

## Context

The base ships a Think-based org agent (`OrgAgent` + `OrgChat` Durable Object pattern, brought in by [[ALW-218]]) but it has drifted and is only half-wired:

- **Think is 11 releases behind.** `packages/agent/package.json` pins `@cloudflare/think@0.8.5`; upstream `origin/main` is at **0.12.1** (siblings also stale: `@cloudflare/shell` 0.3.9 → 0.4.1, `agents` 0.15.0 → 0.17.3). The 0.9→0.12 window landed changes that directly touch how we wire the model, sessions, tools and UI.
- **Multi-session is stubbed, not real.** `OrgAgent.onBeforeSubAgent` only ever spins up one hardcoded `"default"` chat; the `chat_meta` table is built to hold many rows but nothing creates a second.
- **Tool surface is catalog-only.** `tools/compose-org-tools.ts` wires just `createProductTools` + `createDocumentTools` (+ display tools). The agent can read/write **catalog** ERP data but has no reach into the transaction core (payments, wallet, settlement — [[ALW-354]]/[[ALW-355]]), people, or platform/settings. Whether that's correct needs a deliberate decision, not drift.
- **Filesystem exists but is unverified in the UI.** `OrgAgent` holds a `Workspace` (`@cloudflare/shell`); `OrgChat` proxies to it via `SharedWorkspace`. There is no confirmation the file surface is usable/visible to the operator, and 0.12.1 fixes attachment metadata surviving the sub-agent hop.
- **Agent UX/UI has never had a pass.** Streaming, tool-call rendering, and approvals are as-ported. 0.9→0.12 shipped `@cloudflare/think/react` (`useAgentChat` wrapper), `think studio`/`think state` dev CLIs, durable `action()` approvals (incl. dashboard/no-socket approval), and improved tool-call lifecycle hooks.

### Multi-session — the recommended pattern (verified against upstream)

Our parent-directory + per-chat-sub-agent shape is **not a hack — it is exactly the pattern Cloudflare recommends**, now formalized in `design/rfc-think-multi-session.md` ("Multi-Session via Composition"): one Think DO = one conversation; a parent directory agent owns the chat index and shared state; child chats are sub-agents (facets); the browser WebSocket connects **directly to the child**. They explicitly **rejected** "many `Session`s inside one Think DO" (`SessionManager`) because a DO is single-threaded and would serialize a user's chats. So the composition we have is the blessed direction; what's wrong is only our *implementation* of it.

The `Chats` base class from that RFC is **not shipped yet** (`packages/think/src/chats.ts` does not exist at origin/main — status "proposed"); the current concrete reference is **`examples/assistant`** (`agents/assistant/agent.ts` = `AssistantDirectory`, `shared-workspace.ts` = the child proxy). Against that reference our version needs to change:
- **Registry becomes the source of truth for chat existence** — `subAgent()` / `deleteSubAgent()` / `listSubAgents(Child)`; `chat_meta` is pure decoration (title/preview). We hardcode `"default"` and never list.
- **`onBeforeSubAgent` gates, does not create** — return 404 for unknown facets (`!hasSubAgent`), and create via an explicit `createChat()` `@callable`. Ours creates the default inline (the anti-pattern).
- **Shared workspace stays a parent-owned `Workspace` + child `SharedWorkspace` RPC proxy** — the RFC lists "shared `Workspace` across chats" as an explicit **out-of-v1 follow-up with no first-class API**, and `examples/assistant` shares it exactly the way we do (parent owns the real `Workspace`; children forward the whole `WorkspaceFsLike` surface over DO-RPC). So keep the proxy; it is current best practice.
- **Adopt the `Workspace` `onChange` hook** — the directory broadcasts a `workspace-change` event to every connected client so file-browser UI stays live across chats/tabs (`workspaceRevision` in the example's `useChats`). We have nothing like this.
- **Security note to carry:** a shared workspace means any tool in any chat can read/write every file the org owns — fine for a personal assistant, but for an RBAC'd ERP this needs a deliberate gating decision.

Grounding: upstream is checked out at `~/Development/research/agents-cloudflare` (remote `cloudflare/agents`); the per-version delta is in `packages/think/CHANGELOG.md` there. Related draft: [[ALW-348]] (codemode/durable tool approvals) overlaps AC-5 and should be reconciled or folded in.

## Scope

- Upgrade `@cloudflare/think` 0.8.5 → 0.12.1 and align sibling `@cloudflare/shell` / `agents`; absorb the API deltas.
- Adopt the 0.12 model API where it simplifies our wiring.
- Re-align multi-session to the `examples/assistant` reference: registry-authoritative chats, gate-only `onBeforeSubAgent`, explicit create/list/rename/delete, replacing the hardcoded `default` chat.
- Keep the parent-owned shared workspace + child `SharedWorkspace` proxy; add the `onChange`→broadcast so file UI stays live; verify/expose it in the UI, including attachments across the sub-agent hop.
- Audit and right-size the agent tool set (ERP data + platform), keeping the money/document mutation invariant.
- A first UX/UI pass over the chat surface (streaming, tool-call rendering, approvals).
- Capture findings + any deferred follow-ups as sub-tasks.

Touch points: `packages/agent/**` (`org/org-agent.ts`, `org/chat/org-chat.ts`, `org/chat/shared-workspace.ts`, `tools/compose-org-tools.ts`, `inference/chat-models.ts`), the web chat surface under `apps/web/**`, and `packages/agent/package.json`.

## Acceptance criteria

- [ ] AC-1: `@cloudflare/think` is upgraded to 0.12.1 (siblings `@cloudflare/shell`/`agents` aligned to compatible versions); `pnpm typecheck`, `pnpm test`, and `pnpm build` are green and an org chat still completes a turn end-to-end. verify: run the three gates + exercise the chat in `pnpm dev`.
- [ ] AC-2: The org chat model is wired through the 0.12 `getModel()` surface — either returning a model-id string via the built-in resolver or an explicit `LanguageModel` — with `resolveOrgChatModel()`/`getModel()` in `inference/chat-models.ts` + `org-chat.ts` updated accordingly (no dead resolution scaffolding). verify: read the two files; model resolves once per turn.
- [ ] AC-3: Real multi-session, aligned to the `examples/assistant` reference — a user can create, list, switch between, rename, and delete **multiple** named chats in one org; the sub-agent **registry** (`listSubAgents`/`subAgent`/`deleteSubAgent`) is the source of truth and `chat_meta` is decoration; `OrgAgent.onBeforeSubAgent` gates unknown facets with a 404 and no longer hardcodes `default`. verify: create 2 chats in the UI, message each, confirm isolation + persistence across reload; confirm no `"default"` special-case remains.
- [ ] AC-4: The org's **single shared** workspace is confirmed correct and visible: files written from chat A are readable from chat B (parent-owned `Workspace` + child `SharedWorkspace` proxy retained); the `Workspace` `onChange` hook broadcasts a change signal so a workspace-backed UI refreshes live across chats/tabs; attachments sent into a chat survive the sub-agent hop (0.12.1 fetchMetadata fix). verify: write a file in one chat, see it in another + in the file UI without reload; send an attachment, confirm it renders.
- [ ] AC-5: The agent tool set is audited and documented: current coverage enumerated, the intended ERP-data-vs-platform reach decided and implemented (or explicitly deferred as sub-tasks), and every write tool still passes the RBAC `assertCan` guard. New Think capabilities (lifecycle hooks, `@cloudflare/think/tools/fetch`, durable `action()` approvals) are evaluated and either adopted or explicitly declined; reconcile with [[ALW-348]]. The shared-workspace RBAC exposure (any chat tool can touch every org file) is explicitly decided, not left implicit. verify: a short tool-audit note in the PR + `compose-org-tools.ts` reflects the decision.
- [ ] AC-6: A first agent-UX pass is landed — chat streaming, tool-call rendering, and approval prompts are correct and not visually broken; evaluate `@cloudflare/think/react`, the example's `useChats()` sidebar shape, and `think studio` for adoption. verify: walkthrough of a multi-tool, approval-gated turn + switching between two chats.
- [ ] AC-7: Findings and any out-of-scope follow-ups (adopting the `Chats` base class once it ships, multi-session hardening, deeper tool packs, voice/messenger channels, chat-recovery tuning) are written up and filed as linked sub-tasks rather than silently dropped. verify: doc note + linked tasks.

## Non-goals

- Non-web channels (voice/messenger via `configureChannels()`) — note only, do not build.
- Exposing money/document **mutation** tools to the agent (see Boundaries).
- Building new capability packs' tools beyond what the audit deems core.
- Adopting a library `Chats` base class in this task — it is not shipped upstream yet; mirror `examples/assistant` and leave the swap as a follow-up.
- Multi-tenant/perf hardening of the DO beyond making multi-session correct.

## Boundaries

- ✅ Always keep the RBAC `assertCan(...)` guard on every agent write tool (`tools/guard.ts`).
- ✅ Always ground the upgrade + multi-session shape in the real upstream at `~/Development/research/agents-cloudflare` (`packages/think/CHANGELOG.md`, `design/rfc-think-multi-session.md`, `examples/assistant`), not guesses.
- ⚠️ Ask first before wiring a new Cloudflare resource (e.g. an **AI Gateway** binding for the 0.12 string-model gateway routing, or a new `AI` binding) — same ADR-0010 ask-first rule as [[ALW-312]].
- ⚠️ Ask first before a breaking change to the chat persistence schema (`chat_meta` / transcript tables) that would need a migration.
- 🚫 Never expose money or document **mutations** as agent tools — only low-stakes catalog writes are agent-callable; money/document changes hand off to the real screen (existing invariant, see the comment in `tools/guard.ts`).
- 🚫 Never let the agent bypass org scoping (all tools stay bound to `AgentToolsContext` org/user).
