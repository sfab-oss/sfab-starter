# ALW-347 Role-based access control spine (owner/admin/operator) — gate sensitive actions via better-auth

## Context

The starter ships better-auth's organization plugin with a `member.role` column (`owner`/`admin`/`member`) already in the schema (`packages/db/src/schema/auth.ts:51`), but **role is only enforced client-side** in the invite form (`apps/web/src/components/organization/members/invite-member-form.tsx:32`). There is **no server-side authorization** beyond "logged-in + has active org" (`apps/web/src/hono/middleware/auth.ts` — `requireAuth`/`requireActiveOrg`). Any member can today call any org-protected endpoint.

The starter UX plan (Transaction Core surfaces, [[ALW-299]]) introduced money + document operations that must be role-gated — most concretely, **over-limit fiado sales**: everyone sees the warning, an admin can bypass it ("Acepto/Continuar"), a lower role is hard-blocked. The same gate logic governs agent writes, money reversals, and org administration. Because these gates change what each role **sees and can do**, RBAC is a **prerequisite** for the Transaction Core surfaces — building them first means retrofitting permission checks into finished UI. Hence this task **blocks [[ALW-299]]**.

**Design decisions locked with the owner (2026-06-20):**
- **Role-rank, not a fine-grained permission matrix.** Fine-grained resource×action config is a known maintenance/UX nightmare in ERPs. Ship a simple rank `owner > admin > operator` ("operator" = better-auth's `member`, UI copy only — no schema change). **No end-user permission-config UI.**
- **Fixed roles.** better-auth `dynamicAccessControl` stays **off** — orgs assign people to the fixed roles, they cannot invent roles at runtime.
- **Single `can(action, ctx)` seam.** Every gate routes through one helper (role-rank semantics under the hood) so call sites read `can("payment:reverse")`, not `role === "admin"`. Keeps the option to swap in better-auth's resource+action AC later without churning call sites. better-auth already provides `hasPermission` server-side and `checkRolePermission` client-side (network-free) — prefer building on those over a hand-rolled engine.

## Scope

- Add a **`can(action, ctx)` authorization helper** with role-rank semantics (`owner=3 > admin=2 > operator=1`), usable from both server (Hono/core) and client (UI show/disable/explain). Build on better-auth's access-control primitives where practical.
- Add a **server-side guard** (e.g. `requireRole`/`requirePermission` Hono middleware alongside `requireAuth`/`requireActiveOrg`) that reads the caller's `member.role` for the active org and enforces it **before the route handler** — closing the current server-side gap.
- Gate these **four** sensitive actions in v1 (operator blocked / admin+ allowed):
  1. **Credit-limit bypass** — over-limit fiado: operator hard-blocked; admin+ gets the warning with an "Acepto/Continuar" bypass action. (Ties into the Transaction Core sale/finalize flow.)
  2. **Agent low-stakes writes** — the agent's in-chat "Approve" for catalog/inventory edits only fires if the caller's role may make that edit; money/documents always hand off to the real screen (never executed from chat).
  3. **Money reversals / voids** — reversing a payment, voiding a finalized document: admin+ only.
  4. **Org settings & member management** — org settings, invite/remove member, change a member's role: admin+ only (enforce server-side, not just in the invite form).
- **UI affordance pattern:** gated controls render with an honest disabled/explained state for operators (not hidden-and-broken); admin bypass surfaces a confirm dialog ("Acepto/Continuar") for the credit case.
- Rename `member` → **"operador"** in user-facing copy (role value stays `member` in the DB to avoid a migration unless trivial).

## Acceptance criteria

- [ ] AC-1: A `can(action, ctx)` helper exists with role-rank semantics and is the single decision point for gating; call sites reference named actions, not raw role strings. verify: read the helper + grep call sites for `role ===`.
- [ ] AC-2: A server-side guard enforces role for org-protected routes before the handler; an operator calling a gated endpoint gets 403. verify: middleware code + a test hitting a gated route as operator.
- [ ] AC-3: Credit-limit bypass is role-gated — operator is hard-blocked on an over-limit fiado sale; admin+ sees the warning and can confirm "Acepto/Continuar" to proceed. verify: exercise the sale flow as each role.
- [ ] AC-4: Agent low-stakes catalog write via in-chat approve only succeeds when the caller's role may make that edit; money/document execution from chat remains forbidden for all roles. verify: agent approve flow as operator vs admin.
- [ ] AC-5: Money reversal/void and org-settings/member-management are admin+ only, enforced server-side. verify: tests for each as operator (403) and admin (allowed).
- [ ] AC-6: dynamicAccessControl is NOT enabled; there is no end-user permission-config UI; roles are the fixed owner/admin/operator set. verify: auth config + UI review.
- [ ] AC-7: Gated UI controls render an honest disabled/explained state for operators rather than failing silently. verify: UI review as operator.
- [ ] AC-8: `pnpm typecheck && pnpm lint:check && pnpm test` pass. verify: run them.

## Non-goals

- Fine-grained per-resource permission matrices or an end-user role/permission editor.
- Org-defined custom roles at runtime (better-auth `dynamicAccessControl`) — explicitly deferred.
- Teams, additional role tiers (viewer/cashier/accountant) — deferred; revisit if a pilot needs them.
- Designing or building the Transaction Core surfaces themselves — that is [[ALW-299]]; this task provides the gate they consume.

## Boundaries

- ✅ Always: build on better-auth's organization access-control (`hasPermission`/`checkRolePermission`) rather than a parallel hand-rolled engine; enforce on the server, not just the client.
- ✅ Always: keep the role set fixed and the gate logic behind one `can()` seam.
- ⚠️ Ask first: any schema/migration change (the role column already exists; prefer copy-only rename); adding role tiers beyond owner/admin/operator.
- 🚫 Never: enable dynamicAccessControl; ship an end-user permission matrix UI; gate purely client-side; change the sfab platform repo.
