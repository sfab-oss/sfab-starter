# ALW-489 delete-product tool: silent no-op on unknown id + agent name-vs-ULID resolution

## Context

Found twice during [[ALW-480]] browser QA (evidence: workspace `active/alw-480-destructive-approval-verify/artifacts/browser-qa/alw480-20260710-approval/`). When the org agent calls `delete_product` with an id that matches no row (it passed the product NAME, e.g. `"QA480-C"`, instead of the ULID), the tool returns `{status: "completed", executionId: …}` and the assistant reports success — but nothing was deleted. Approval mechanics themselves are fine (ALW-456 verified green); this is a tool-contract and id-resolution problem. A destructive tool that claims success without effect erodes trust in an AI-native ERP, even though it fails safe.

## Scope

- Make `delete_product` (and audit sibling mutating tools) fail loudly when 0 rows are affected: error result, not `completed`.
- Improve id resolution: tool description/schema should force ULID ids (the agent resolved the ULID correctly in only 1 of 4 flows), or the tool should resolve unambiguous name→id itself and error on ambiguity.

## Acceptance criteria

- [ ] AC-1: `delete_product` with a non-existent id returns an error surfaced in the chat card (not Completed). verify: browser, delete by wrong id
- [ ] AC-2: Asking the agent to delete a product by name reliably deletes it (correct ULID resolved or name resolved server-side) across repeated runs. verify: browser, 3 repetitions
- [ ] AC-3: 0-row-affected audit covers the other mutating catalog tools (update/create paths) with the same loud-failure contract. verify: code + tests

## Non-goals

- Approval-flow changes (verified green in ALW-480); pretty card labels ([[ALW-401]]).

## Boundaries

- ✅ Always: local seed data only.
- 🚫 Never: prod keys or remote wrangler.
