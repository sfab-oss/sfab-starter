# ALW-354 Transaction Core settlement engine — payments, allocations, projections & conservation (base)

## Context

Second track of the Transaction Core implementation, **split out of [[ALW-299]] on 2026-07-01** during planning review: ALW-299 now delivers the money foundation + the `documents`/`line_items` hub + folio-atomic finalize + the de-verticalized neutral demo; **this task builds the settlement engine on top of that hub**. It implements `docs/architecture/transaction-core.md` §4/§5/§10 (the merged ALW-328 design — consume its shapes, do not re-decide them) **minus the `customer_credit` wallet**, which is deferred to its own task (no base-demo consumer yet; fiado = plain AR and works without a wallet).

**Engineering constraints (inherited from the ALW-299 planning review — binding):**
- **D1 has no interactive transactions.** Production D1 rejects `BEGIN/COMMIT`; `db.transaction()` passes local miniflare but fails on real D1. Every multi-write mutation is a **single `db.batch([...])`** of pre-assembled statements; do all reads before batch assembly (accept TOCTOU; the full-scan `rebuild*` model is the correctness backstop). `db.transaction()` is banned in review.
- Tests that touch D1 run through the existing **`apps/web` workers-pool harness** (`@cloudflare/vitest-pool-workers` + `readD1Migrations`). Verify = full `pnpm typecheck && pnpm lint:check && pnpm test`.

## Scope

- **`payments` table:** signed `amount`, open `method` (no enum), `paidAt`, `reference`, `idempotencyKey`, `reversesPaymentId`.
- **`payment_allocations` table:** `paymentId`↔`documentId`↔`amount`, `effectiveAt`, `reversedAt`, **`unique(orgId, paymentId, documentId)`** (red-team **C4**).
- `contract` inbound Zod for payment/allocation inputs; **allocation UPSERT semantics explicit** — re-allocating the same payment→doc pair is an UPDATE under the unique index, not a second row (AI-review forward note a).
- **`core.recordPayment({header, allocations[]})`** as one `db.batch()`; allocations target **fiscal documents only**.
- **Projections:** `amountPaid`/`balanceDue`/`paymentStatus`/`entity.balance` are derived; first-class full-scan **`rebuildDocumentPayment`/`rebuildEntityBalance`** (`lastAppliedPaymentId` = perf hint only); **reversals = compensating rows** (red-team **C6**); consume the reserved `payment:reverse` RBAC key.
- **Conservation (reduced, wallet-less form):** enforce in `core` — `Σ allocations against a doc = doc.amountPaid`, and `entity.balance = Σ open fiscal-sales balanceDue` (red-team **C1** reduced form; the `− creditBalance` term arrives with the wallet task). `entity.balance` is **sales-fiscal AR only** — supplier `bill`s (AP) never poison it (red-team **C2**).
- **`credit_note` disposition routing:** implement **cash refund** and **apply-to-document**; the `disposition` enum ships complete but the **store-credit branch returns a clear not-implemented error** until the wallet task lands.
- **`creditLimit` enforcement** on fiado (AR) sales + consume the `credit:bypass` RBAC key.
- Define + test **exactly when `sale_completed` fires vs `document_finalized`** (AI-review forward note c — payment-dependent, so it lives here).
- Extend the neutral demo: **record payment → balance updates**; partial payments; fiado (unpaid finalized invoice raises entity balance).
- Tests: double-pay, AP isolation, reverse-old-payment-then-rebuild, re-allocation = UPDATE not a second row, partial fan-out leaves the predecessor open, credit-note/void draw their own folios (never reuse).

## Acceptance criteria

- [ ] AC-1: `payments` + `payment_allocations` exist per `transaction-core.md` §4/§5, including the `unique(orgId, paymentId, documentId)` index. verify: schema + migration snapshot.
- [ ] AC-2: `recordPayment` (and every multi-write mutation here) is a single `db.batch()`; no `db.transaction()` anywhere in the diff. verify: `grep -rn "db.transaction" packages apps` returns nothing.
- [ ] AC-3: Projections are derived with full-scan `rebuild*`, and reversals are compensating rows. verify: reverse-an-old-payment test rebuilds to correct balances.
- [ ] AC-4: Conservation enforced in `core`: Σ allocations = `amountPaid`; `entity.balance` = Σ open fiscal-sales `balanceDue`; AP `bill`s excluded. verify: named tests (double-pay, AP isolation).
- [ ] AC-5: Allocation upsert semantics are explicit in `contract`; re-allocating the same payment→doc pair updates the existing row. verify: test asserts one row.
- [ ] AC-6: `credit_note` `disposition` routes cash-refund and apply-to-document end-to-end; store-credit is a clear, tested not-implemented error. verify: tests for all three branches.
- [ ] AC-7: `sale_completed` vs `document_finalized` firing conditions are defined and tested (they differ for unpaid fiado finalize).
- [ ] AC-8: `payment:reverse` and `credit:bypass` RBAC keys are consumed (no longer dead reservations). verify: read `packages/auth/src/access-control.ts` call sites.
- [ ] AC-9: The demo shows finalize → pay → balance; `pnpm typecheck && pnpm lint:check && pnpm test && pnpm build` pass. verify: run all four + walkthrough.
