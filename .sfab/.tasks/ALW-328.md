# ALW-328 Transaction Core architecture + the primitives around it (Tier-1)

## Context

Tier-1's **hub design**, done **after** the scaffold ([[ALW-326]]) so it lands in the decided directory structure ([[ALW-327]]). Transaction Core is the hub the whole Mexican pack-set (POS, CFDI, fiado, …) grafts onto, and POS is its forcing function — so it earns careful design (ADR-0010 §E/§F; ADR-0017). This is **design + decision**; the output is documented architecture. **Implementing** Transaction Core (migrations, services, UI) is a *separate* build task.

## Scope

### Transaction Core (the hub)
- The Documents + LineItems model (quote = order = invoice = PO, one structure); decimal-safe money; the generic tax field; document numbering/folios; the status lifecycle.
- The **domain-event spine** it emits (`sale_completed`, …) — the shared backbone feeding pack integration (the hook), the event log (BI), and the activity timeline (ADR-0010 §F).

### Surrounding primitives
- How the surrounding primitives relate to the hub: Catalog (line items reference products), Entity & Contact (counterparty + running balances), Activity & Audit Log (events + history), File Hub (attached documents).

### Seams
- The **seams** it exposes for packs (ADR-0017; cf. [[ALW-299]]): optional 1:1 compliance-doc relation, open `paymentMethod`, etc. — additive (ADR-0010 §D generosity over seams).

**Output:** a `docs/architecture/transaction-core.md` (and/or an ADR) — design, not full implementation.

## Acceptance criteria

- [ ] AC-1: A Transaction Core architecture doc defines the Documents+LineItems model, decimal-safe money, the generic tax field, document numbering/folios, and the status lifecycle. verify: read `docs/architecture/transaction-core.md`.
- [ ] AC-2: The doc defines the domain-event spine the hub emits and how it feeds the pack hook, the event log (BI), and the activity timeline. verify: read the doc + cross-check ADR-0010 §F.
- [ ] AC-3: The doc shows how Catalog, Entity & Contact, Activity & Audit Log, and File Hub relate to Transaction Core. verify: read the doc.
- [ ] AC-4: The doc lists the seams the hub exposes for packs. verify: cross-check ADR-0017 / [[ALW-299]] seams.

## Non-goals

- **Implementing** Transaction Core (schema migrations, services, UI) — a separate Tier-1 build task.
- The directory/package structure — [[ALW-327]] (this lands in it).
- The other primitives' detailed internal design beyond how they relate to the hub.
- The registry / pack mechanism — [[ALW-324]] / ADR-0017.

## Boundaries

- ✅ Always: domain-neutral naming; design to the ADR-0010 §E parts list + §F event spine; *legible, not optimal* (§D stopping line).
- ⚠️ Ask first: introducing a model that diverges from the §E parts list.
- 🚫 Never: couple the schema to a vertical/country; over-engineer a plugin framework (generosity over seams, ADR-0010 §D).
