# ALW-453 Fix chat attachment send — provider rejects non-text content types

## Launch cut (2026-07-10)

**Status: soft stretch** under [[ALW-479]]. In scope for launch polish. Prefer fix; if provider remains text-only, **hide/disable attach** with clear UX rather than leaving a broken control. Not a hard gate if text chat works and attach is gated.

## Context

During ALW-427 file-handling QA on the ALW-451 branch (PR #53), attaching an image (or file) in org-agent chat succeeds in the UI (chip + remove), but **sending** the message fails with:

`messages.content.type is invalid, allowed values: ['text']`

Provider under test: `ORG_CHAT_PROVIDER=zai-coding-plan` (Z.AI). Text-only messages work. This blocks AC-3 attachment round-trip.

## Scope

- Trace how attachment parts are serialized into the model request (PromptInput files → OutgoingMessage parts → OrgChat / AI SDK messages).
- Either convert/upload attachments into a form the active provider accepts, **or** gate unsupported attachment types / hide the attach control with a clear pre-send error (no opaque API error card).
- Confirm image and common document types against the configured provider(s); document any provider that is text-only.

## Acceptance criteria

- [ ] AC-1: Attaching a PNG and sending produces a successful assistant turn (or a clear unsupported-type toast / attach disabled before send) — verify: local chat with `zai-coding-plan`

- [ ] AC-3: Text-only send remains green; no regression of ALW-451 DO-name fix
