# ALW-530 Wide-dialog create + detail-page edit for product and entity forms

## Context

Catalog and entity create flows use small dialogs that feel cramped once fields (and images) pile up. Edit already lives on the detail page, which is the right host, but form craft is inconsistent across surfaces. Operators need a clear create vs edit pattern: create opens a roomy dialog from the list; edit stays on `$id`.

## Scope

- Replace **create product** and **create entity** small dialogs with a **wide dialog** (same fields as today; more horizontal room — not a side sheet).
- Keep **edit product** and **edit entity** on the **detail page**; improve form craft there to match the create dialog (shared form component where natural).
- Shared craft for these forms: dense labels, RHF + zod + inline field errors, clear primary/cancel actions, dirty-gate on save (disable submit when clean).
- Align empty-list / header “New …” CTAs so they open the wide create dialog.

## Acceptance criteria

- [ ] AC-1: From catalog list, “New product” (or equivalent) opens a wide create dialog; submitting creates the product and closes or navigates to detail as today. verify: browser on `/catalog`
- [ ] AC-2: From entities list, “New …” opens a wide create dialog with the same create fields; submit works. verify: browser on entities list
- [ ] AC-3: Product and entity edit remain on their `$id` detail pages (no edit dialog). verify: open existing product/entity → edit in page
- [ ] AC-4: Create and edit for product share one form implementation (props for mode); same for entity — no duplicated field trees. verify: code review
- [ ] AC-5: Forms use RHF + zod with visible field errors; save disabled when not dirty on edit. verify: leave a field invalid; edit without changes
- [ ] AC-6: Typecheck/lint/tests green for touched packages. verify: `pnpm` typecheck/lint/test on web

## Non-goals

- Side sheets / slide-overs as the create host.
- Document/invoice create or line-editor redesign (follow-up).
- Record-payment / collect flow redesign (follow-up).
- Settings form rewrite; verb-first nav / Hoy dashboard.
- New fields beyond what create/edit already expose.

## Boundaries

- ✅ Always: reuse existing `ProductForm` / entity form patterns; keep money/catalog write policy; Spanish copy where those surfaces already use it.
- ⚠️ Ask first: changing post-create navigation (stay on list vs open detail).
- 🚫 Never: invent sheets; touch document payment UI in this PR; push to `main`.
