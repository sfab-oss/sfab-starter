# ALW-324 Build the capability-pack registry — registry.json, code-bearing packs, ephemeral install skills (ADR-0017)

## Context

ADR-0010 **Decision 2** made `sfab-starter` itself the shadcn **GitHub-repo registry**, strictly ref-coupled. **ADR-0017 re-scopes the registry from UI blocks to *code-bearing capability packs*** — each pack a whole vertical slice (**Drizzle schema + server code + UI + a `skill.md`**) that an agent installs via `shadcn add sfab-oss/sfab-starter/<pack>#<ref>`. This task **builds that mechanism**.

This is the **Tier-2** layer, sequenced **after** the base scaffold/primitives ([[ALW-299]] exposes the base *seams* packs graft onto; [[ALW-312]] owns component *authoring*) and **after** the [[ALW-327]] D10 directory layout lands. The mechanism is designed but not built — no `registry.json` exists today. Registry-of-one for v1; promotion/triage is out of scope, gated on [[ALW-170]].

Starting point: `packages/ui` is already a shadcn registry *consumer* (`@ai-elements` → `registry.ai-sdk.dev`); nothing makes the repo a *producer*.

## Scope

- Root `registry.json` declaring **packs**. shadcn registries ship arbitrary files (`registry:lib/hook/page/file` alongside `registry:ui/block`) — confirm the item-type taxonomy at build.
- `apps/registry/` — the **showcase app** that hosts packs in its workspace, doubles as the **Layer-1 test harness**, and runs the **build step** emitting `registry.json`. The app is dropped at fabrication; its workspace stays for dev.
- Pack-folder layout: the **layer-sliced** `registry/<pack>/{db, contract, core, server, tools, components, skill.md, *.test.ts}` per ADR-0017 §6 + the [[ALW-327]] D10 layout (mirrors the base package structure — `db`/`contract`/`core`/`server`/`tools`/`components` — NOT the old flat `{schema, server, components}`).
- The **ephemeral install skill** convention (ADR-0017 §2): a one-shot, install-time procedure that wires the pack into the base seams, **writes a provenance manifest entry, then deletes itself**.
- The **provenance manifest** (ADR-0017 §3): extend `.sfab/template.json` with `packs: {<name>: {ref, installedAt}}` — **no `mode` flag**; the installed `ref` is the only stored divergence baseline.
- Prove the path end-to-end with **one small, additive code-bearing pack** (schema + a seam + an install skill). The flagship **POS** pack is its own later task.
- Document consumption: `shadcn add sfab-oss/sfab-starter/<pack>#<ref>`, ref-coupling, the public-only constraint, and the install-skill + manifest convention.

## Acceptance criteria

- [ ] AC-1: Root `registry.json` validates and the `apps/registry` build step emits it. verify: `pnpm --filter registry build` (or `npx shadcn registry:build`) succeeds.
- [ ] AC-2: One **code-bearing** pack laid out per the D10 layer-sliced layout (`db/contract/core/server/tools/components` + `skill.md`) is declared and resolves to its source files + `registryDependencies`. verify: inspect the built pack JSON against [[ALW-327]] D10.
- [ ] AC-3: `shadcn add sfab-oss/sfab-starter/<pack>#<ref>` pulls the pack into a fresh project and its install skill wires it. verify: run in a scratch project (or document the dry-run + skill steps).
- [ ] AC-4: Installing the pack writes `packs.<name> = {ref, installedAt}` to `.sfab/template.json`, with **no `mode` field**. verify: inspect the manifest after install.
- [ ] AC-5: Layer-1 — the pack is imported/rendered/unit-tested in `apps/registry`. verify: `pnpm --filter registry test`.
- [ ] AC-6: README documents the pack model, `#<ref>` coupling, the install-skill + manifest convention, and the public-only constraint. verify: read README.
- [ ] AC-7: `pnpm typecheck && pnpm lint:check && pnpm test && pnpm build` pass. verify: run all four.

## Non-goals

- The flagship **POS** pack — its own later task (this proves the mechanism with a small pack).
- The **Layer-2 install-eval harness over sandboxes** (dogfood) — platform-side, separate.
- Promotion intake (Issues → triage → tasks) + multi-template **registry-of-many** — gated on [[ALW-170]].
- Component *authoring* of hard-parts — [[ALW-312]]; base *seams* — [[ALW-299]].
- Cross-repo bump propagation — [[ALW-306]].
- A private registry / custom registry server.

## Boundaries

- ✅ Always: packs import from `@workspace/ui` aliases + tailwind tokens; release on the same tag as the rest of the template (one-tag coupling); keep `sfab-oss/sfab-starter` public.
- ✅ Always: install skills are ephemeral (self-delete) and write the provenance entry; the manifest stores `{ref, installedAt}` only; pack folders follow the [[ALW-327]] D10 layer slices.
- ⚠️ Ask first: changing how the template is tagged/released; adding the Layer-2 sandbox harness here.
- 🚫 Never: store a forked/tracked `mode` flag; couple a pack to a vertical's domain naming; assume a custom registry server in v1; revert to the flat `{schema, server, components}` layout.
