# ALW-324 Publish sfab-starter as a shadcn registry (registry.json + ref-coupled blocks)

## Context

ADR-0010 **Decision 2** decides that `sfab-starter` is itself the shadcn **GitHub-repo registry**: a `registry.json` at the repo root declares shadcn-compatible component **blocks**, and fabricated projects pull them with `shadcn add sfab-oss/sfab-starter/<block>#<ref>`, **strictly coupled** to the exact template ref they're pinned to. The mechanism is **designed but not built** — no `registry.json` exists today.

This is the **publication layer**, distinct from [[ALW-312]], which owns the component *cleanup* and authoring the hard-part composables (markdown editor, PDF viewer, Excel viewer). This task takes whatever blocks the UI package exposes and makes them installable into a pinned project; ALW-312 decides *what* the blocks are.

Starting point already in the repo: `packages/ui` ships a working shadcn setup — `components.json` (new-york, lucide, the alias block) that is already a registry **consumer** (`@ai-elements` → `registry.ai-sdk.dev`) plus the base shadcn components. There is no `registry.json` making `sfab-starter` a registry **producer**.

Per ADR-0010 this is **registry-of-one** for v1; the promotion/triage pipeline (Issues → triage skill → tasks) is explicitly out of scope and gated behind [[ALW-170]].

## Scope

- Author a root `registry.json` valid against the shadcn registry schema, declaring `sfab-starter`'s installable blocks.
- Define at least one real block end-to-end so the path is proven; blocks import from the project's `@workspace/ui` aliases + tailwind tokens (so they only resolve against the matching base ref — the intended strict coupling).
- Add the shadcn registry **build/validate** step (e.g. `registry:build`) that emits the per-block JSON the GitHub-repo path serves, wired into the existing turbo/scripts setup.
- Keep `components.json` aliases/`registries` consistent with the produced registry.
- Document consumption in the README: `shadcn add sfab-oss/sfab-starter/<block>#<ref>`, the ref-coupling rule, and the **public-only** constraint (the GitHub-repo registry path requires a public repo).

## Acceptance criteria

- [ ] AC-1: A root `registry.json` exists and validates against the shadcn registry schema. verify: `npx shadcn registry:build` (or schema validate) succeeds; file present at repo root.
- [ ] AC-2: At least one block is declared and resolves to its source files + `registryDependencies`. verify: inspect the built block JSON output.
- [ ] AC-3: `shadcn add sfab-oss/sfab-starter/<block>#<ref>` against a pinned ref pulls the block into a consumer app and it compiles. verify: run the command in a scratch app (or document the dry-run output).
- [ ] AC-4: README documents the registry, the `#<ref>` coupling, and the public-only constraint. verify: read README.
- [ ] AC-5: `pnpm typecheck`, `pnpm lint:check`, `pnpm test`, `pnpm build` pass. verify: run all four.

## Non-goals

- Promotion intake (GitHub Issues → triage skill → tasks) and multi-template **registry-of-many** — deferred per ADR-0010, gated behind [[ALW-170]]. v1 is registry-of-one.
- Authoring the hard-part components themselves (markdown editor, PDF viewer, Excel viewer) — that's [[ALW-312]]; this task publishes whatever blocks exist.
- A private registry / custom registry server (would break the public GitHub-repo path).

## Boundaries

- ✅ Always: blocks import from `@workspace/ui` aliases + tailwind tokens; release `registry.json` on the same tag as the rest of the template (one-tag coupling, ADR-0010).
- ✅ Always: keep `sfab-oss/sfab-starter` public (the registry path depends on it).
- ⚠️ Ask first: adding a CI/release step that changes how the template is tagged.
- 🚫 Never: couple a block to a vertical's domain naming; assume a custom registry server in v1.
