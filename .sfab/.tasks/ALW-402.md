# ALW-402 Org-agent: broaden read-only tool reach (transaction core / people / settings)

## Context

Deferred from [[ALW-398]] AC-5 (org-agent tool audit, decision "keep lean, defer breadth"). Today the org agent's ERP reach is catalog-only: read/write **products** + **read** documents (`packages/agent/src/tools/compose-org-tools.ts`), all writes gated by `assertCan`. It has no visibility into the transaction core ([[ALW-354]] settlement, [[ALW-355]] wallet), people, or org settings — so it can't answer "what does this customer owe?" or "what's their store-credit balance?" even read-only.

This task adds **read-only** reach only. The money/document **mutation** invariant is untouchable: those changes hand off to the real screen and never become agent tools (see the comment in `tools/guard.ts`).

## Scope

- Add read-only agent tools over the transaction core: payments/allocations, wallet balance + ledger, settlement projections/conservation views.
- Add read-only people/entities + org-settings tools as useful for grounding.
- Wire them through `getOrgAgentTools` (codemode `tools.*`); keep display tools separate.
- Right-size descriptions so the model picks the correct read tool; add per-tool tests.

## Acceptance criteria

- [ ] AC-1: The agent can read transaction-core state (payments, wallet balance/ledger, settlement projection) for its org, scoped by `AgentToolsContext`. verify: tool unit tests + a chat that answers a balance question in `pnpm dev`.
- [ ] AC-2: Read-only people/entities + settings tools exist and are org-scoped. verify: tests.
- [ ] AC-3: No money or document **mutation** tool is added; `tools/guard.ts` invariant intact; any write still passes `assertCan`. verify: grep the tool set; review.

## Non-goals

- Any money/document **mutation** tool.
- Role-based per-tool authorization beyond the existing `assertCan` (that is [[ALW-400]]).
- Web UI.

## Boundaries

- ✅ Always keep every tool bound to `AgentToolsContext` org/user scoping.
- 🚫 Never expose money/document mutations as agent tools.
