# ALW-312 Transform sfab-starter into the AI-native ERP golden base (manual, comment-driven)

## Context

The single umbrella task for the **manual** transformation of `sfab-starter` into the golden, AI-native ERP base the fabricate pipeline ([[ALW-306]]) instantiates. This is product/UX work done **by hand, step by step, human-in-the-loop** — progress is captured as **comments on this task**, not split into many subtasks (for now). The golden codebase is the reference every fabricated app starts from, so quality (UI/UX/schema/code) is the bar.

Distinct from the platform machinery in epic [[ALW-306]]. A few scoped starter tasks already exist and stay standalone: [[ALW-299]] (generalize the vertical), [[ALW-298]] (rename to sfab-starter), [[ALW-111]] (migration guard), [[ALW-112]]/[[ALW-109]] (DB knowledge/reviewer), [[ALW-304]] (Cloudflare Email). This task covers the not-yet-ticketed work below.

## Scope (worked manually; each becomes its own comment thread, possibly spun out later)

- **ERP scope-plan (first):** define what a generic-enough ERP includes — employee management, file/image uploads, financials, lists — standardized so it's later swappable to another business context via AI. Plan up front, then build step by step; the UI/UX must look nice.
- **UI component registry:** clean it up; live in a separate package/app, **omitted when fabricating a project** (no bloat) but kept in the template. Don't re-wrap what shadcn already composes well — add the **hard parts** with clean APIs: markdown editor, PDF viewer, Excel/spreadsheet viewer, etc. Give the AI more freedom on basics (layout), pre-built composables for the hard parts.
- **Live documentation system:** establish format/templates, how it's maintained, where it lives, how versioning is tracked (ties to ADR-0010 Decision 5 / capability docs).
- **Golden-codebase cleanup:** de-"vibe-code" the existing parts to the quality bar.
- **Cloudflare-native integrations (without overdoing it):** **AI Gateway** (model observability/caching) is the clear win; email is already [[ALW-304]]. Add value, not resources-for-their-own-sake.

## Acceptance criteria

- [ ] AC-1: An ERP scope-plan exists and is agreed (captured as a comment / doc).
- [ ] AC-2: The component registry is cleaned, lives separately, is fabricate-omitted, and includes the agreed hard-part components.
- [ ] AC-3: The live-documentation format + maintenance + versioning approach is established.
- [ ] AC-4: AI Gateway is integrated into the template.
- [ ] AC-5: The codebase passes the golden-quality bar (lint/typecheck/tests green; structure clean).

## Non-goals

- The fabricate/deploy pipeline machinery (epic [[ALW-306]]).
- The already-ticketed scoped starter work ([[ALW-299]], [[ALW-298]], [[ALW-111]], [[ALW-304]], …) — those stay standalone.
- Multiple templates / the component-registry-of-many (ADR-0010 reaffirms registry-of-one for v1).

## Boundaries

- ✅ Always: domain-neutral naming (ADR-0010 amendment); keep capabilities (AI assistant, search, uploads) tested + documented.
- ⚠️ Ask first: adding a new Cloudflare resource type to the base (it grows the provisioning surface in [[ALW-306]] #5).
- 🚫 Never: couple the worked example to a single vertical's naming; bloat fabricated apps with the registry viewer.
