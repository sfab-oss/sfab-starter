# ALW-350 Build the first capability pack (POS) + complete the pack-install mechanism (ADR-0017)

## Context

[[ALW-324]] landed the capability-pack registry *foundation* — the repo as a `registry.json` **producer**, the single shadcn `RegistryItem` primitive discriminated by `meta.sfabKind`, the docs gallery + Layer-1 harness, and the drift-checked build step (`packages/registry`). But it merged with **every item as `sfabKind: \"block\"`**: the `pack` value is a defined-but-dead seam, and the actual capability-pack mechanism (code-bearing packs, ephemeral install skills, provenance) was **deferred** rather than built.

This task completes that mechanism and proves it end-to-end with the flagship **POS** pack as the first real vertical — POS doubles as the proving pack ALW-324's ACs called for. The `meta.sfabKind: "pack"` branch is the seam already in place to graft onto.

Sequenced **after** [[ALW-299]] (the hub + folio-atomic finalize + `shouldAffectStock` gate + seam-marker) and [[ALW-354]] (the settlement engine — POS records payments). **[[ALW-355]] (the wallet) is NOT a blocker**: POS v1 sells cash + fiado (plain AR); returns-to-store-credit lights up once the wallet lands. Built on [[ALW-327]] D10 layer-sliced layout (done).

**This task also owns two base-side pieces deliberately deferred from ALW-299 (2026-07-01 planning review), because the pack is their first consumer and should shape them:**
1. **The critical-handler extension mechanism** — how a pack contributes a statement to `finalize`'s atomic `db.batch()`. ALW-299 leaves an explicit statement array + a seam-marker comment; this task designs the actual mechanism (per ADR-0017 likely an **install-skill source edit** at the marker — NOT a runtime `registerCriticalHandler` plugin registry; "not a plugin framework"). Every multi-write stays one `db.batch()` — `db.transaction()` is banned (fails on production D1).
2. **The stock-decrement critical handler + its atomicity test** — the pack's `movements` table + a stock decrement that commits-or-rolls-back **with** finalize, gated by the base's `shouldAffectStock(line)`; the test proves finalize + folio + stock succeed or fail together (the flagship §7/§8 scenario from `transaction-core.md`).

## Scope

- A code-bearing **POS** pack under `packages/registry/registry/pos/` following the [[ALW-327]] D10 layer slices: `db / contract / core / server / tools / components` + `skill.md` + `*.test.ts`.
- The **server** slice as a **Hono route** (the starter's Workers stack) — not a Next.js handler.
- The pack's **`movements` table + stock critical handler** wired into `finalize`'s batch at ALW-299's seam marker, gated by `shouldAffectStock` (see Context #2).
- The **critical-handler extension mechanism** itself, designed with this first consumer (see Context #1).
- Wire `meta.sfabKind: "pack"` through the registry generator + types so a pack item builds into `registry.json` with its full multi-layer file set + `registryDependencies` (today only `block` items are exercised).
- The **ephemeral install skill** (ADR-0017 §2): a one-shot install-time procedure that wires the pack into the base seams (including the finalize-batch edit), writes the provenance entry, then **deletes itself**.
- The **provenance manifest** (ADR-0017 §3): extend the fabricated-project `.sfab/template.json` (see `.sfab/template.example.json` + `.sfab/README.md` — the live file exists only in fabricated projects) with `packs: {<name>: {ref, installedAt}}` — **no `mode` flag**; the installed `ref` is the only stored divergence baseline.

## Non-goals

- The **Layer-2 install-eval harness over sandboxes** (dogfood) — platform-side, separate.
- Promotion intake + multi-template **registry-of-many** — gated on [[ALW-170]].
- Any pack beyond POS; cross-repo bump propagation ([[ALW-306]]).
- A private / custom registry server.
- Returns → store credit (needs [[ALW-355]]'s wallet; POS v1 = cash + fiado/AR).

## Boundaries

- ✅ Always: the pack imports from `@workspace/ui` aliases + tailwind tokens; releases on the same tag as the rest of the template (one-tag coupling); keep `sfab-oss/sfab-starter` public; install skills are ephemeral (self-delete) and write the provenance entry; `db.batch()` for every multi-write.
- ⚠️ Ask first: changing how the template is tagged/released; adding the Layer-2 sandbox harness here; making the extension mechanism a runtime plugin registry (the spec leans install-time source edit).
- 🚫 Never: `db.transaction()`; store a forked/tracked `mode` flag; couple the pack to a vertical's domain naming in the *mechanism*; assume a custom registry server in v1; revert to the flat `{schema, server, components}` layout.
