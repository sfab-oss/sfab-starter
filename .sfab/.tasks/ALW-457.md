# ALW-457 Multi-language support with AI-assisted translation

## Context

The starter is English-only in practice (empty `packages/core/i18n` skeleton, hardcoded `lang="en"`), with known Spanish RBAC drift in org members/settings. Target users include Spanish-speaking operators; launch content also wants a "multi-language + agent bootstrap" beat. We need real catalogs + switcher + an **agent-friendly** sync/lint loop — not one-off string fixes.

**Settled (2026-07-11 sharpen):** EN-default bilingual; cookie preference; app-only v1; agent-primary translation; ship docs + agent skill.

## Scope

- Adopt **Paraglide JS** (TanStack Start–aligned) for message catalogs, cookie locale strategy, SSR `html lang`, language switcher.
- Catalogs: `en.json` source of truth + `es.json` target; glossary for ES; hash lockfile so only new/changed EN strings need retranslation.
- Scripts: `pnpm i18n:sync`, `pnpm i18n:lint` (human + `--json` gap report). Agent fills target locales from lint gaps (no API keys required in public starter/CI for v1).
- Cover app chrome: shell/nav (verb-first Spanish labels when locale=`es` where they fit), auth/onboarding, settings, catalog, documents/money loop, org-agent **UI chrome** (not model replies). Wire `formatMoneyMinor` / dates to active locale.
- Persist locale via **cookie** (per device); default `en`; never silently flip English users to Spanish.
- Fix mixed-locale RBAC copy (invite-member Spanish toast/hints, etc.) onto catalogs.
- Ship **`docs/guides/i18n.md`** and a repo **`.agents/skills/…` skill** covering: add a locale (e.g. French), keep catalogs current while developing, catalog-first (never hardcode UI strings).
- Link skill from template-init / AGENTS.md as appropriate.

## Acceptance criteria

- [ ] AC-1: EN + ES catalogs ship with documented layout/load path (Paraglide/inlang). verify: inspect message files + build.
- [ ] AC-2: User can switch locale via UI; preference persists across reload (cookie); primary chrome + one core flow render in selected locale; default remains `en`. verify: browser walkthrough.
- [ ] AC-3: `i18n:sync` and `i18n:lint` (incl. `--json`) exist; lint fails on missing/empty targets or placeholder mismatch; incremental story documented (hash lockfile or equivalent). verify: run sync/lint; force a gap and see failure.
- [ ] AC-4: Agent workflow documented: skill + `docs/guides/i18n.md` cover add-locale and develop-as-you-go; agent can complete ES/new-locale gaps from lint output without a vendor API key. verify: read skill + guide; dry-run add-locale steps.
- [ ] AC-5: Known mixed-locale strings (incl. invite-member Spanish toast/RBAC hints) are on the i18n path — no hardcoded second-language leftovers in English builds. verify: grep + switcher smoke.
- [ ] AC-6: Browser walkthrough evidence for locale switch on shell + one core flow (catalog or invoice). verify: walkthrough artifact.

## Non-goals

- Translating LLM/org-agent model output or user-generated content.
- Org-level or DB-backed locale preference (cookie only in v1).
- URL prefixes (`/es/...`) in v1.
- Localizing `packages/registry` demo blocks (follow-up task).
- Paid TMS (Crowdin/Tolgee) or required CI LLM Action with secrets.
- Full professional localization QA / legal-certified translations.
- RTL languages.
- Flipping the shipped default locale to Spanish.

## Boundaries

- ✅ Always: English is catalog source of truth; cookie default `en`; sync/lint are mechanical (Biome-shaped); ship guide + agent skill; evidence the switcher.
- ⚠️ Ask first: adding a third shipped locale beyond ES; introducing URL-based locale routing; making `i18n:translate` LLM script required; org-level locale.
- 🚫 Never: silently change default locale for existing English users; commit API keys or translation-vendor secrets; hardcode user-visible UI strings outside catalogs in new code; translate model replies in this task.
