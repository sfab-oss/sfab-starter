# ALW-457 Multi-language support with AI-assisted translation

## Context

The starter app is English-only today, with known locale drift (e.g. a hardcoded Spanish toast flagged in [[ALW-428]]). Target users include Spanish-speaking operators; shipping real multi-language support (not one-off string fixes) is a product gap. AI-assisted translation is the preferred way to keep locale catalogs current without a full human translation pipeline for v1.

## Scope

- Pick and wire an i18n approach for the starter UI (message catalogs, locale detection/selection, formatting for dates/numbers/currency where the app already surfaces them).
- Cover user-visible chrome and flows: shell/nav, auth/onboarding, settings, catalog, documents/money loop, org-agent chat UI chrome (not model replies).
- AI-assisted translation path: generate/update locale catalogs from the English source (script, CI step, or agent workflow) with a clear review/override story for critical copy.
- Language switcher (or equivalent) so an org/user can choose locale; persist the preference.
- Fix existing mixed-locale copy as part of adopting the system (including the Spanish toast seed).

## Acceptance criteria

- [ ] AC-1: At least two locales ship (English + one other, Spanish preferred) with a documented catalog layout and load path.
- [ ] AC-2: User can switch locale and the preference persists across reload; primary chrome and the main product flows render in the selected locale.
- [ ] AC-3: AI-assisted translation workflow exists and is documented (how to regenerate catalogs from English, what is reviewed vs auto-accepted).
- [ ] AC-4: Known mixed-locale strings (including the invite-member Spanish toast) are on the i18n path — no hardcoded second-language leftovers in English builds.
- [ ] AC-5: Browser walkthrough evidence for locale switch on shell + one core flow (e.g. catalog or invoice).

## Non-goals

- Translating LLM/org-agent model output or user-generated content.
- Full professional localization QA / legal-certified translations.
- Per-org custom translation overrides beyond a simple locale preference (unless a later task).
- RTL languages in v1.

## Boundaries

- ✅ Always: keep English as the source of truth for catalogs; document the AI translation workflow; evidence the switcher.
- ⚠️ Ask first: which second locale(s) beyond Spanish; whether preference is user-level vs org-level; which i18n library/stack.
- 🚫 Never: silently change default locale for existing English users; commit API keys or translation-vendor secrets into the repo.
