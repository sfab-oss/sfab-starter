# ALW-567 Template init: tell adopters to delete .sfab

## Context

Public template / clone adopters still get the starter's `.sfab/` tree (platform task mirrors, fabrication manifests). When the factory fabricates a project, that folder is stripped automatically. For the GitHub template path there is no automatic scrub — agents and humans need an explicit delete step so they do not treat platform scaffolding as product code.

## Scope

- Add a clear early step in `docs/template-init.md`: delete `.sfab/` when adopting from the public template; explain factory strips it, template does not, and it is not needed for their product.
- Add a short pointer in README Getting started so agents see it before deep reshaping.
- Optional one-line cross-link from `AGENTS.md` / `.claude/CLAUDE.md` if it stays in the adopt path index without expanding day-to-day conventions.

## Acceptance criteria

- [ ] AC-1: `docs/template-init.md` instructs adopters/agents to remove `.sfab/` early, with factory-vs-template rationale. verify: read the file
- [ ] AC-2: README Getting started mentions deleting `.sfab/` (or points at template-init for that step). verify: README.md
- [ ] AC-3: Copy does not tell people to edit `.sfab/` contents; delete-only. verify: wording

## Non-goals

- Changing fabrication / factory drop lists; scrubbing `.sfab` from this repo's own history; gitignore changes for the template repo itself.

## Boundaries

- ✅ Always: keep guidance in agent-facing adopt docs; explain why (factory auto-strips).
- ⚠️ Ask first: deleting `.sfab` from the published template repo so clones never see it.
- 🚫 Never: hand-edit `.sfab/.tasks/*` or `.sfab/.docs/*` as part of this change; invent factory behaviour that is not true.
