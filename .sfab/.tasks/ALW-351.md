# ALW-351 Build the on-brand registry gallery — Phase-0 spine first, hero/file surfaces deferred (UI-only)

## Context

[[ALW-324]] shipped the registry **mechanism** + 3 seed items (`button`, `badge`, `dashboard-01`): each item is a directory under `packages/registry/registry/<name>/` with an `item.ts`, previewed inline (`registry:ui`) or full-screen-in-an-iframe (`registry:block`) in the docs gallery (`/view/$name`). The skeleton works; it has almost nothing in it.

This task **fills the registry** with the canonical, on-brand building blocks for the golden ERP base, authored as reviewable gallery blocks with **mock data (UI-only, no backend)**. The owner reviews each block in the gallery, says "this is wrong here / here," and we iterate until it's on-brand. The result is a registry of trusted, on-brand parts; the **real golden app is then assembled FROM these reviewed blocks** in [[ALW-299]] — the gallery is the design/review surface, not the app.

**Sequencing decision (owner, 2026-06-21, after a 4-voice review):** do **less up front**. Ship a tight **Wave 1 = the Phase-0 spine** (~14 blocks: the primitives, the list spine, the shell honesty pass) and **defer** the heavy hero/money-document surfaces, the file viewers, and mobile chrome to **Wave 2**, started only after Wave 1 is reviewed and signed off. Building Wave-2 surfaces before the Wave-1 primitives are frozen would cement wrong money/table/sheet patterns and create a divergent "prettier fiction" parallel ERP — the panel's unanimous top risk.

The target design is already ratified. Binding, committed grounding a contractor must read:
- `docs/architecture/operator-ux.md` — §3 composition spine, §4 route/sheet map, §5 surfaces, §6 RBAC, §8 build phases, §9 decisions D1–D43.
- `docs/architecture/transaction-core.md` + `docs/decisions/006-transaction-core.md` — the capabilities every money/document surface maps to.
- The existing repo before authoring anything new (reuse > rebuild): `packages/ui/src/components/brand/data-table.tsx` (already filter/sort/columns/pagination), `.../app-layout.tsx`, `.../app-breadcrumbs.tsx`; `apps/web/src/components/search/search-command.tsx` (⌘K already wired on `cmdk`); `apps/web/src/components/catalog/product-form.tsx` (the RHF+Zod+`Field` form convention); `apps/web/.../inventory/index.tsx:251` (the USD `Intl` hack to kill). `packages/ui/src/components/shadcn/button.tsx` is the canonical authoring pattern (`cn`/`cva`/`Slot.Root`/`data-slot`; pure `ui`, never imports `core`).

## Scope

- Author blocks under `packages/registry/registry/<name>/` with **mock data** + `item.ts`, drift-gate green (`pnpm --filter @workspace/registry generate`). Set `type` (inline vs iframed) + `meta` per item.
- **Reuse > extend > build, deliberately.** Where the starter already ships it, the registry item is a **showcase/reskin**, not a rebuild (verified against the repo by the panel — see tags below).
- New **primitives** go in `@workspace/ui` (pure, no `core`); domain **composites** in the block/app layer.
- **Responsive** (tablet-primary, phone supported, desktop secondary — operator-ux §9).
- **Land incrementally** — one block (or one small group) per PR — so review is continuous, never one mega-PR. Prefer **merged, richer blocks** over many tiny previews of the same spine.
- **Fix the seed `dashboard-01`**: its four fake revenue KPIs + English copy contradict operator-ux §5.1 — reskin into the proper `app-shell` + honest `hoy-overview` (Spanish verb-first nav).

## Wave 1 — Phase-0 spine (this task's committed scope; author + review these first)

Tags: [reuse]=showcase existing · [extend]=grow a real base · [build]=new. Build order top→bottom.

8. **hoy-overview** [build] — replaces the fake dashboard: PrimaryActionBar + ≤3 honest KPI tiles + action items + capped movements feed (no charts).
9. **app-shell** [reuse/reskin] — compose existing `AppLayout` + collapsible sidebar + breadcrumbs + org-switcher + agent dock; reskin to **Spanish verb-first nav**; introduce a shared **`navigation-config`** (one source for sidebar + ⌘K). Replaces `dashboard-01`'s shell.

11. **side-sheet-host** [build] — one global `SideSheetHost` + `UrlSheet` with a **payment-sheet mock** (URL-addressable `?sheet=…`); contact/wallet as nested variants, not separate blocks.
12. **confirm-ladder** [mix] — one block showing all tiers: Sonner toast (T0) · AlertDialog (T1) · TypeNameConfirm stub (T2) · **TotalConfirm** (T3, fast-sale exact-total). Reuse shipped `sonner`/`alert-dialog`.

Cross-cutting in Wave 1: an **RBAC disabled-affordance** pattern (operator sees disabled+explained controls, mock `can()`); agent dock + org-switcher appear only as **reuse demos inside app-shell**, not re-authored.

- The capability-**pack** mechanism — [[ALW-350]].

- Standalone `button`/`badge` demos (keep only as trivial reference), `bulk-action-bar`, global entity/doc search in ⌘K (would be faked) — cut.

## Boundaries

- ✅ Always: reuse/extend the shipped primitives before building; blocks import from `@workspace/ui` + tailwind tokens; pure primitives in `@workspace/ui`, domain composites in the block layer; mock data only (labeled); responsive (tablet-primary); follow the `button.tsx` authoring idioms.
- ⚠️ Ask first: starting any **Wave 2** item before Wave 1 sign-off; promoting/deferring a whole group; adding any backend dependency; deviating from an operator-ux decision (D1–D43).
- 🚫 Never: present fake data as real *in the app* (gallery mock is fine, labeled); fork `@workspace/ui` primitives; re-author a shipped component the panel marked [reuse]; add a charts/notification surface the plan cut; build the real app wiring here.
