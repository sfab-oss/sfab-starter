# ALW-351 Build the on-brand registry gallery — author the canonical "one of everything" blocks (UI-only) for the golden ERP base

## Context

[[ALW-324]] shipped the registry **mechanism** + 3 seed items (`button`, `badge`, `dashboard-01`): each item is a directory under `packages/registry/registry/<name>/` with an `item.ts`, previewed inline (`registry:ui`) or full-screen-in-an-iframe (`registry:block`) in the docs gallery (`/view/$name`). The skeleton works; it has almost nothing in it.

This task **fills the registry** with the canonical, on-brand building blocks for the golden ERP base — **one canonical instance of every big UI piece** ("one of everything"), authored as reviewable gallery blocks with **mock data (UI-only, no backend)**. The owner reviews each block in the gallery, says "this is wrong here / here," and we iterate until it's on-brand. The result is a registry of trusted, on-brand parts; the **real golden app is then assembled FROM these reviewed blocks** in [[ALW-299]] — the gallery is the design/review surface, not the app.

The target design is already ratified. Binding, committed grounding a contractor must read:
- `docs/architecture/operator-ux.md` — §3 composition spine (primitives + composites), §4 route/sheet map, §5 per-surface designs, §6 RBAC, §8 build phases, §9 decisions D1–D43.
- `docs/architecture/transaction-core.md` + `docs/decisions/006-transaction-core.md` — the capabilities every money/document surface maps to.
- `packages/ui/src/components/*` — the existing primitives to compose from; `packages/ui/src/components/shadcn/button.tsx` is the canonical authoring pattern (`cn`/`cva`/`Slot.Root`/`data-slot`; pure `ui`, never imports `core`).

(Owner-side provenance, local-only / not committed: the 2026-06-20 UX swarm under `docs/.local/2026-06-20-1222-starter-ux-design/` — `GALLERY-PLAN.md`, `COMPONENT-PIECES.md`, `UX-PLAN.md`. The catalog below is distilled from it so this ticket is self-contained.)

## Scope

- Grow the registry from 3 seed items to the **canonical gallery set** below, authored as `registry:block` / `registry:ui` items under `packages/registry/registry/<name>/`, each with **mock data** + an `item.ts` + a gallery preview, drift-gate green (`pnpm --filter @workspace/registry generate`).
- **Reuse vs build, deliberately:** plain shadcn/`@workspace/ui` primitives that already exist (Button, Input, Badge, Dialog…) get a **lightweight showcase only** — no bespoke block. The effort goes into the **value-add** pieces: the data table + filtering, the money/document surfaces, and the file viewers (PDF / markdown) the owner specifically wants.
- Build the underlying **primitives** the blocks need in `@workspace/ui` (pure) and **composites** in the block layer (domain-aware) — respect the layer rule (no `core` in `ui`).
- **Responsive** (tablet-primary, phone supported, desktop secondary — operator-ux §1/§9 decision).
- Land **incrementally** — one block or one group per PR — so the owner reviews continuously, never one mega-PR.

## Candidate item inventory (author one canonical instance each; [reuse]=showcase only, [build]=author/extend)

Build order follows operator-ux §8 / fastest review signal.

- [reuse/extend] App shell block: per-route header + collapsible icon sidebar + breadcrumbs (seed `dashboard-01` already shows this — promote to the canonical shell block).
- [build] Command palette ⌘K (Ir a + Acciones; shares `navigation-config`; RBAC-disabled actions show reason).
- [build] Mobile bottom tab bar (Hoy · Vender · Cobrar · Inventario · Más).

- [build] `FilterPresetBar`: URL-synced preset chips + debounced search + "Más filtros" popover.
- [build] `CollectionStateSet`: honest skeleton / empty+CTA / preset-empty / error+retry / stale.
- [build] `MasterDataListPage` spine (header + FilterPresetBar + ResourceTable).
- [build] Hoy overview block: PrimaryActionBar + ≤3 KPI tiles + action items + capped movements feed (no charts).

**3 — Detail, forms & field primitives**
- [extend] `RecordPageShell` + `PropertyGrid` (full-page record read surface).
- [reuse] `ContractForm` pattern (`zodResolver(contract/<cap>)` + RHF `Field`) — showcase the convention.

**4 — Overlays, dialogs, sheets & confirms** (one canonical exemplar of each)

**5 — Agent, chat & file viewers** *(includes the owner-requested differentiators)*

These are the high-value "extra stuff" pieces; pdf + markdown + image first.

## Per-block review loop

1. Author the block under `packages/registry/registry/<name>/` with mock data + `item.ts`, set `type` (`registry:ui` inline vs `registry:block` iframed) + `meta` (sfabKind `block`, iframeHeight for blocks).
2. `pnpm --filter @workspace/registry generate` → it appears in the gallery and full-screen at `/view/<name>`.
3. Owner reviews in the gallery, leaves per-area feedback.
4. Iterate until the owner marks the block **on-brand**. Land that block/group; move on.

## Non-goals

- **Wiring blocks to the real backend / Transaction Core** — that's [[ALW-299]] (it assembles these reviewed blocks into the real app). UI-only mock data here.
- The capability-**pack** mechanism — [[ALW-350]].
- Authoring the `sfab-component-authoring` skill itself — separate; use it as the *how* once it exists.
- New architecture or UX decisions — operator-ux D1–D43 is binding; this **executes** it, it doesn't re-open it.

## Boundaries

- ✅ Always: blocks import from `@workspace/ui` aliases + tailwind tokens; pure primitives in `@workspace/ui`, domain composites in the block layer; mock data only (labeled); responsive (tablet-primary); follow the `button.tsx` authoring idioms (cn/cva/Slot.Root/data-slot).
- ⚠️ Ask first: promoting/deferring a whole group; adding any backend dependency; deviating from an operator-ux decision (D1–D43); File-Hub storage wiring (viewers are mock here).
- 🚫 Never: present fake data as real *in the app* (gallery mock is fine, labeled); fork `@workspace/ui` primitives; add a charts/notification surface the plan cut; build the real app wiring in this task.
