# ALW-332 Add a working db:seed script for local first-run

## Context

The root `package.json` exposes `db:seed` (`pnpm --filter web db:seed`), but `apps/web` has **no `db:seed` script**, so `pnpm db:seed` errors. The README quickstart had to drop the seed step because of this (see PR #8). A starter should give a first-run user some sample data so they can sign in and immediately see the app and the built-in org agent working, instead of an empty database. A seed helper already exists for tests (`apps/web/test/helpers/seed.ts`: `seedUser`, `seedOrganization`, etc.) and can be reused.

Note: `db:seed:prod` (root → `web db:seed:prod`) is dangling in the same way and should be handled consistently (wire it or remove it).

## Scope

- Add a `db:seed` script to `apps/web/package.json` that seeds the **local** D1.
- Implement a seed entrypoint (reuse/refactor `apps/web/test/helpers/seed.ts`) that inserts a demo user + organization (and any minimum rows needed to sign in).
- Make seeding idempotent (safe to re-run).
- Re-add the `pnpm db:seed` step to the README quickstart, and document the seeded sign-in credentials.
- Reconcile the dangling `db:seed:prod` root script (wire a guarded prod seed or remove it).

## Acceptance criteria

- [ ] AC-1: `pnpm db:seed` completes without error against a freshly migrated local D1. verify: `pnpm db:reset && pnpm db:migrate && pnpm db:seed`
- [ ] AC-2: After seeding, a demo user and organization exist in local D1, and the seeded user can sign in. verify: sign in with the documented seed credentials at `localhost:4011`
- [ ] AC-3: Seeding is idempotent. verify: run `pnpm db:seed` twice with no error and no duplicate rows
- [ ] AC-4: README quickstart re-includes `pnpm db:seed` and documents the seed credentials.
- [ ] AC-5: `db:seed:prod` is either wired (remote/guarded) or removed from root + app `package.json` so no dangling delegation remains. verify: `pnpm db:seed:prod` does not fail with "script not found"

## Non-goals

- Large/fake-data fixtures or a faker dependency; a minimal demo user + org is enough.
- Seeding any remote/production database by default.

## Boundaries

- ✅ Always: keep it local-first; reuse the existing schema and seed helpers; idempotent inserts.
- ⚠️ Ask first: seeding production data; adding new data-generation dependencies.
- 🚫 Never: commit real secrets; run a seed against `--remote`/prod without an explicit, guarded flag.
