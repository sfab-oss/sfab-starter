# ALW-401 Org-agent UI/UX — multi-session sidebar, file browser & tool-call/approval rendering

## Context

The org agent's backend is modernized and multi-session-capable in [[ALW-398]] (Think 0.12.1, registry-authoritative chats with `createChat`/`listChats`/`renameChat`/`deleteChat` RPC, a shared workspace with an `onChange` broadcast, attachments across the sub-agent hop). This task builds the **web chat surface** that consumes it — the part the operator actually sees and uses.

Today the chat UI (under `apps/web/**`) is single-session and has never had a UX pass: no way to manage multiple chats, no view of the agent's filesystem, and as-ported tool-call/approval rendering. The upstream reference for the client shape is `examples/assistant/src` (`use-chats.ts` sidebar hook + `workspaceRevision`, `client.tsx`), and `@cloudflare/think/react` ships a Think-tuned `useAgentChat`.

Depends on [[ALW-398]] for the RPC surface; do not start the multi-session sidebar / file browser until that lands.

## Scope

- Multi-session **sidebar**: list, create, switch, rename, delete chats, wired to the `OrgAgent` directory RPC from [[ALW-398]]; per-tab active-chat selection (client-side).
- **File browser** for the shared workspace: list/read files the agent has written, refreshing live off the `workspace-change` broadcast / `workspaceRevision` signal (no polling). Reuse the existing registry viewers where they fit (markdown/pdf/spreadsheet) for file preview.
- **Tool-call + approval rendering**: show tool calls, inputs/outputs, and approval prompts clearly; a denied/guarded action reads as a clean message, not a broken state.
- **Streaming polish** and general chat-surface UX (loading, errors, reconnect).
- Evaluate adopting `@cloudflare/think/react` (`useAgentChat`) and the `useChats()` sidebar shape vs. our current client wiring.

Touch points: `apps/web/**` (chat routes/components), possibly a thin client hook mirroring `examples/assistant/src/use-chats.ts`. Reuse `packages/ui` + registry viewer components; no backend/DO changes (those are [[ALW-398]]).

## Acceptance criteria

- [ ] AC-1: An operator can create, switch between, rename, and delete multiple named chats from a sidebar; state persists across reload and different tabs can hold different active chats. verify: walkthrough — create 2 chats, message each, rename one, delete one, reload.
- [ ] AC-2: A file browser shows the agent's shared-workspace files and updates live when the agent writes/deletes a file in any chat (driven by the `workspace-change`/`workspaceRevision` signal, not polling); at least text/markdown files can be previewed (reusing registry viewers where applicable). verify: have the agent write a file in chat A, see it appear in the browser while viewing chat B, without reload.
- [ ] AC-3: Tool calls render with their name, input, and result; an approval-gated action shows an approval prompt and a role-denied action surfaces as a clean in-chat message (no unhandled error / broken UI). verify: walkthrough of a multi-tool, approval-gated turn.
- [ ] AC-4: Streaming, loading, error, and reconnect states are handled gracefully on the chat surface. verify: walkthrough incl. an interrupted/reconnected turn.
- [ ] AC-5: A decision is recorded on whether to adopt `@cloudflare/think/react` / the `useChats()` shape, with the chosen client wiring in place. verify: note in the PR + the client code reflects it.

## Non-goals

- Any backend / Durable Object / RPC changes — owned by [[ALW-398]].
- Role-based authorization UI (denied-by-role affordances beyond a clean message) — that's [[ALW-400]].
- Non-web channels (voice/messenger).
- Bloating the registry viewers or coupling them to the agent (reuse only).

## Boundaries

- ✅ Always reuse existing `packages/ui` + registry components (shell, viewers) rather than introducing parallel UI.
- ✅ Always keep the chat surface domain-neutral (ADR-0010) — no vertical-specific copy.
- ⚠️ Ask first before adding a new client dependency beyond `@cloudflare/think/react` / the `agents` React hooks already in the stack.
- 🚫 Never surface money/document mutation actions as agent-executable in the UI — those hand off to the real screen (mirrors the backend invariant).
