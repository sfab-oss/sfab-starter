# ALW-411 Org-agent inference: Cloudflare AI Gateway + env-driven pluggable provider registry (Vercel / Workers AI / GLM)

## Context

Today `OrgChat` and `OrgSubAgent` resolve their model through the **Vercel AI SDK global `gateway()`** (`google/gemini-3-flash`) in `packages/agent/src/inference/chat-models.ts`, with the key read ambiently from `AI_GATEWAY_API_KEY`. There is **no Cloudflare AI Gateway** and no way to swap the underlying provider. [[ALW-312]] AC-4 (Cloudflare AI Gateway) was carried under the now-merged golden-base umbrella but never shipped; this task extracts and finishes it, broadened.

We want **Cloudflare AI Gateway as an always-available top layer** (caching, analytics, rate-limiting, retries) sitting in front of a **pluggable underlying provider selected by environment variable**: the Vercel AI Gateway (current default), Cloudflare **Workers AI**, or an OpenAI-compatible **subscription** such as z.ai **GLM** (coding plan). This mirrors the main sfab platform's provider abstraction (`packages/contract/src/model-registry.ts` + `.../inference/chat-models.ts`: a `Provider` union → data-only `AI_SDK_PROVIDER_BUILD` map → single `buildAiSdkModel(offering, apiKey)` factory), but with **keys sourced from environment variables**, not the D1 `provider_keys` table (the platform's DB+AES path is out of scope here).

Architecture note (verified against Cloudflare docs 2026-07): CF AI Gateway fronts any OpenAI-compatible HTTPS upstream via a **Custom Provider** (`gateway.ai.cloudflare.com/v1/{acct}/{gw}/custom-{slug}/…` or the `/compat` unified endpoint) with a `cf-aig-authorization` header — so "CF on top, provider underneath" is a single integration with a swappable `baseURL` + key. Routing a non-CF-native upstream (GLM, or the Vercel gateway itself) requires a one-time custom-provider created in the CF dashboard/API (config, not code).

## Scope

- Introduce, in `packages/agent/src/inference/`, an env-driven provider registry mirroring the platform shape:
  - a `Provider` union (`vercel-gateway | workers-ai | zai-coding-plan`, extensible);
  - a data-only `AI_SDK_PROVIDER_BUILD` map: `{ kind: "gateway" } | { kind: "openai-compatible", baseURL, providerName }`;
  - a `MODEL_OFFERINGS` catalog carrying `entryId`, context-window / compaction limits (folding the current `MODEL_CONTEXT_WINDOWS` table), and optional per-model `providerOptions`;
  - a single `buildAiSdkModel(offering, apiKey)` factory: `createGateway({apiKey})(id)` for `gateway`, `createOpenAICompatible({name, baseURL, apiKey}).chatModel(id)` for `openai-compatible`, wrapped with `wrapLanguageModel` + `defaultSettingsMiddleware` when `providerOptions` are set.
- Thread `env` through `resolveOrgChatModel(env)` and pass `apiKey` **explicitly** (Workers-correct; stop relying on ambient env). Update `org-chat.ts` and `org-sub-agent.ts` call sites and their `resolveModel()`.
- Env-based key loader: returns `{ provider, value }` from env — `ORG_CHAT_PROVIDER` selects the provider, `ORG_CHAT_MODEL` overrides the model, per-provider key vars supply the secret. No DB, no encryption.
- Add `@ai-sdk/openai-compatible@^2` to `packages/agent` (paired with `ai@^6`, matching the platform; needed because GLM streams `delta.reasoning_content` that the strict `@ai-sdk/openai` schema drops).
- **Cloudflare AI Gateway wrap:** optional, gated on `CF_ACCOUNT_ID` / `CF_AIG_GATEWAY_ID` / `CF_AIG_TOKEN`. When set, the `openai-compatible` providers route through the gateway (rewrite `baseURL` to the custom/compat endpoint + inject `cf-aig-authorization`); when unset, they hit the origin directly. Document the one-time CF custom-provider setup for GLM / non-native upstreams.
- Providers delivered: `vercel-gateway` (default, `gemini-3-flash` — **no behaviour change**), `zai-coding-plan` (GLM via z.ai coding baseURL, key `ZAI_API_KEY`, models `glm-5.2` etc., thinking via `providerOptions`), `workers-ai` (a `@cf/*` model via the Workers AI OpenAI-compatible endpoint — no `env.AI` binding, runs under `wrangler dev`).
- Update `apps/web/.dev.vars.example`, the `@workspace/env` type declarations, and note the wrangler secrets in docs.

## Non-goals

- DB-backed / per-org provider keys, encryption, and BYOK setup UI — that is the platform's [[ALW-14]]/[[ALW-15]]. **Env vars only** here.
- A model-picker UI or per-request model selection (single org-configured model).
- Double-gatewaying the Vercel gateway (CF in front of Vercel) beyond noting it is possible but redundant.
- Any change to money/document mutation paths.

## Boundaries

- ✅ Always: domain-neutral naming; keep the AI-assistant capability tested; **default (no-new-env) behaviour byte-identical** to today.
- ⚠️ Ask first: adding a Cloudflare `env.AI` (Workers AI) binding or any new Cloudflare resource — prefer the gateway/REST path to avoid growing the provisioning surface ([[ALW-312]] boundary; ties to [[ALW-306]] #5).
- 🚫 Never: commit real API keys; never expose money/document mutations as agent tools; never hand-edit generated env types (`packages/env/src/env.d.ts`, `worker-configuration.d.ts`).
