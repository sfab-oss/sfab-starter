# ALW-301 Add pre-push hook to block direct pushes to main

## Context

The `sfab` platform repo blocks direct `git push` to `main` via `.husky/pre-push`, forcing all changes through PRs. `sfab-starter` has no `pre-push` hook — contributors (or agents) can push straight to `main` unless GitHub branch protection is enabled (not available on private repos without Team).

This is a lightweight, local safety net that pairs with the CI workflow task.

## Scope

- Add `.husky/pre-push` adapted from `sfab/.husky/pre-push` — reject pushes where the remote ref is `refs/heads/main`.
- Keep the helpful error message (create branch → push branch → `gh pr create`).
- Ensure the hook is executable and wired via existing `prepare: husky` in root `package.json`.

## Acceptance criteria

- [ ] AC-1: `.husky/pre-push` exists and blocks `git push origin main`. verify: `git push origin main` on a test commit fails with the guard message.
- [ ] AC-2: `git push -u origin <feature-branch>` still succeeds. verify: push a non-main branch.
- [ ] AC-3: Hook is committed with executable permissions. verify: `test -x .husky/pre-push`.

## Non-goals

- Server-side branch protection rules (GitHub settings — separate, plan-dependent).
- Blocking pushes to other protected branches.
- CI workflow itself — separate task.

## Boundaries

- ✅ Always: copy the proven sfab hook verbatim except path/comments if needed.
- 🚫 Never: add bypass flags or weaken the main-branch guard.
