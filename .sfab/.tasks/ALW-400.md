# ALW-400 Role-based authorization for agent tools & the shared workspace (design + implement)

## Context

The org agent currently authorizes at a **coarse** grain and leaves a known gap:

- Agent **write tools** are gated by a single `assertCan(action, { role })` check in `packages/agent/src/tools/guard.ts`, backed by the `can(...)` matrix in `packages/auth/src/access-control.ts` (the owner/admin/member RBAC spine from [[ALW-347]]). It's all-or-nothing per action — there's no per-tool, per-resource, or per-field policy, and read tools aren't gated at all.
- The agent's **shared workspace** has **no authorization whatsoever**: it is one `Workspace` owned by the parent `OrgAgent`, and every chat's tools reach every file the org owns via the `SharedWorkspace` proxy (see [[ALW-398]] AC-4/AC-5). We are deliberately keeping it fully shared **for now** — this task is where we revisit that.

We explicitly deferred this from the org-agent modernization sweep ([[ALW-398]]): shared workspace stays open, and role-based authorization for tools/files is designed later — here.

This is a **design-first** task: the point is to decide the authorization model before building it, then land it once agreed.

## Scope

- A short design note proposing the authorization model for agent capabilities: granularity (per-tool / per-action / per-resource / per-field), how it composes with the existing `can(...)` matrix, and where it's enforced (tool wrapper vs. inside each tool).
- Role-based gating of the **shared workspace** — decide whether it stays fully shared, gets per-role read/write scoping, or gets path/namespace partitioning; implement the chosen model at the `SharedWorkspace` / parent `Workspace` boundary.
- Extend RBAC to **read** tools and to any transaction-core/platform tools added by [[ALW-398]] AC-5, not just catalog writes.
- Keep the money/document mutation invariant intact (those remain non-agent-callable regardless of role).

Touch points: `packages/agent/src/tools/guard.ts`, `packages/agent/src/tools/compose-org-tools.ts`, `packages/agent/src/org/**` (workspace boundary), `packages/auth/src/access-control.ts`.

## Acceptance criteria

- [ ] AC-1: A design note captures the chosen authorization granularity and enforcement point, and how it extends (not forks) the `can(...)` / role matrix in `packages/auth/src/access-control.ts`. verify: the note is in the PR / docs.
- [ ] AC-2: The shared-workspace authorization decision is made and implemented — fully-shared (documented as intentional), per-role read/write, or path-partitioned — enforced at the `SharedWorkspace`/parent `Workspace` boundary, with a test proving the chosen policy. verify: workspace access test + code at the boundary.
- [ ] AC-3: Every agent tool (read and write) is gated by the authorization layer; an operator/member role is denied an action it shouldn't perform and the denial surfaces cleanly in chat (not an unhandled error). verify: a tool-authorization test per role + a walkthrough of a denied action.
- [ ] AC-4: The money/document-mutation invariant still holds — no such mutation is agent-callable at any role. verify: assertion/test + `compose-org-tools.ts` review.

## Non-goals

- Reworking the human-facing (non-agent) RBAC from [[ALW-347]] beyond what the agent layer needs.
- Multi-tenant authorization within one org directory.
- Exposing money/document mutations as agent tools under any role.

## Boundaries

- ✅ Always extend the existing `can(...)` role matrix rather than inventing a parallel authorization system.
- ✅ Always keep every agent write tool behind an authorization check (`assertCan` or its successor).
- ⚠️ Ask first before adding new roles or changing the owner/admin/member model from [[ALW-347]].
- 🚫 Never let the agent bypass org scoping or the money/document-mutation invariant, at any role.
