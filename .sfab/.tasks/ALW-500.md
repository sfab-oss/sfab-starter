# ALW-500 Invalidate React Query after agent tool mutations

## Context

Org agent tools mutate via `@workspace/core` on the Durable Object. React Query invalidation today only runs in UI mutation hooks (`useUpdateProduct.onSuccess`, etc.). After an agent tool updates a product/document/entity, the open page stays stale until a manual refetch.

Query key factories already exist (`getProductKey`, `getProductsKey`, …). Think's `useAgentChat` `onToolCall` is for client tools/approval — not a generic server-tool → cache bridge. Need a clean generic path.

## Scope

- Add a client-side observer (or equivalent) that reacts to completed agent tool parts / results and invalidates the matching React Query keys.
- Prefer a registry keyed by tool name → queryKey factories (reuse exports from `use-products` / `use-entities` / `use-documents`).
- Cover create/update/delete for catalog, entities, documents at minimum.
- Verify: agent updates a product while detail/list is open → UI refreshes without navigation.

## Acceptance criteria

- [ ] AC-1: Agent `update_product` (or codemode equivalent) on an open product detail refreshes the form/fields without reload — verify: browser
- [ ] AC-2: Agent create/delete invalidates list queries so catalog/entities/documents tables update
- [ ] AC-3: Invalidation path is generic (registry or shared helper), not one-off per tool UI card
- [ ] AC-4: No false invalidation storms on read-only tools (`get_product`, `list_products`, display tools)

## Non-goals

- Optimistic UI for agent tools; rewriting tools to go through HTTP; page-context changes.

## Boundaries

- ✅ Always: reuse existing `get*Key` factories; invalidate on successful writes only.
- ⚠️ Ask first: introducing a new WebSocket event type vs watching `helpers.messages` tool parts.
- 🚫 Never: invalidate on every streaming token; couple tool renderers to queryClient ad hoc.
