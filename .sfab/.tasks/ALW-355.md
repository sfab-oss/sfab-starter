# ALW-355 Customer-credit wallet (saldo a favor / anticipos) — complete the base money subgraph

## Context

The final deferred slice of the Transaction Core **base** money subgraph, split out of [[ALW-299]] on 2026-07-01. The wallet **is base architecture** per `docs/architecture/transaction-core.md` §4/§5 — this deferral is about *timing*, not reclassification: it had no consumer in the base demo (the neutral flow is quote → order → invoice → pay; a wallet only appears in overpay/deposit/store-credit edge flows), and its one open design question (walk-in matching) needs a forcing consumer to shape it. Fiado (selling on credit) is plain AR and already works via [[ALW-354]] without a wallet.

First real consumers: POS returns → store credit, anticipos (customer deposits), saldo a favor from overpayment.

**Engineering constraints (binding, same as [[ALW-354]]):** every multi-write mutation is one `db.batch([...])` — `db.transaction()` is banned (fails on production D1); D1-touching tests run through the `apps/web` workers-pool harness.

## Scope

- **`customer_credit` table:** append-only, signed amounts, **nullable `entityId`** for walk-ins (red-team **C3**); compensating rows for corrections (never UPDATE in place).
- **Overpay/deposit routing:** an overpayment or deposit lands **only** in the wallet — never also as an AR allocation (red-team **C1**, full form; closes the double-count hole).
- **Full conservation identity**, upgrading ALW-354's reduced form: `entity.balance = Σ open fiscal-sales balanceDue − creditBalance` — both halves enforced in `core` + full-scan `rebuildCreditBalance`.
- **Walk-in matching strategy — decide + implement here.** Proposed default from the 2026-07-01 planning review: a **claim-reference token** — a walk-in deposit writes a short reference on the `customer_credit` row; redemption requires presenting it and the reference is carried on the allocation. (Session-based matching was considered and rejected as fragile.) Confirm or replace during build; document the decision.
- **`store_credit` branch of `credit_note` `disposition`** — replaces ALW-354's not-implemented error with the real wallet write.
- **Anticipo flow:** deposit → wallet → apply to a later document (redemption reduces `creditBalance` and allocates to the doc in one batch).
- UI/demo surface: show saldo a favor on the entity view; redeem it during payment.

## Acceptance criteria

- [ ] AC-1: `customer_credit` exists per §4/§5 — append-only, signed, nullable `entityId`. verify: schema + a test that corrections are compensating rows.
- [ ] AC-2: A deposit/overpay lands wallet-only and is never double-counted as an AR allocation. verify: anticipo double-count test (C1).
- [ ] AC-3: The full conservation identity (`entity.balance = Σ open fiscal-sales balanceDue − creditBalance`) is enforced in `core` with a full-scan `rebuildCreditBalance`. verify: named conservation test covering both halves.
- [ ] AC-4: The walk-in matching strategy is decided, documented, and implemented with a test (deposit → later redemption by reference). (C3)
- [ ] AC-5: `credit_note` `disposition: store_credit` writes the wallet end-to-end (return → saldo a favor → redeem). verify: test + walkthrough.
- [ ] AC-6: `pnpm typecheck && pnpm lint:check && pnpm test && pnpm build` pass; no `db.transaction()` in the diff.

## Non-goals

- CFDI anticipo **netting** (compliance sidecar) — a CFDI pack concern per ADR-006.
- Gift cards, loyalty points, multi-currency wallets.
- Changing ALW-354's payments/allocations shapes — extend, don't rework.

## Boundaries

- ✅ Always: append-only wallet (compensating rows); `db.batch()` atomicity; consume `transaction-core.md`'s shapes.
- ⚠️ Ask first: replacing the claim-reference walk-in strategy with something else; any new table beyond `customer_credit`.
- 🚫 Never: UPDATE/DELETE on wallet rows; a wallet balance column as source of truth (it is a rebuilt projection); `db.transaction()`.
