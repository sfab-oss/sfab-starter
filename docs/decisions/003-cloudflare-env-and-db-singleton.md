# ADR-003: Shared Cloudflare Env Types & Direct Singleton Pattern

**Status:** Accepted
**Date:** 2026-06-18 (supersedes the 2026-02-26 record; renumbered + rename-corrected)
**Authors:** Team

## Context

`packages/core` is the service layer where domain logic lives. As it grows, every
surface (Hono routes, MCP, AI tools, jobs) imports from it. An earlier factory
pattern (`createDb(d1)`, `createAuth(db, opts)`) required a **wiring file in the
app for every package that needed bindings** — manageable with two consumers,
unworkable as the service layer grows.

Separately, Cloudflare env types (`Cloudflare.Env`: D1 bindings, secrets) were
only available in the app via `worker-configuration.d.ts`; shared packages had no
typed access to `import { env } from "cloudflare:workers"`.

Because the project is committed to Cloudflare Workers as its **only** runtime,
the portability the factory pattern bought was never used.

## Decision

> A shared **`packages/env`** package provides auto-generated `Cloudflare.Env`
> type declarations. Packages **import `env` from `cloudflare:workers` directly**
> and export **singletons** — so any layer does `import { db } from
> "@workspace/db"` with no wiring file. This affirms and extends D5: `core` is
> Workers-runtime code, tested with the workerd test pool.

## Options Considered

### Shared env types + direct singletons (chosen)

`packages/env` exposes generated `Cloudflare.Env`; `db`/`auth` import `env`
directly and export singleton instances.

- **For:** zero DI boilerplate; env types auto-generated from `wrangler.jsonc`
  (no drift); new consumers just import; fewer files in the app.
- **Against:** packages are coupled to the Workers runtime; unit tests need module
  mocking or the workerd test pool, not simple mock injection.

### Keep the factory pattern (rejected)

- **For:** runtime-agnostic packages; trivial mock injection.
- **Against:** a wiring file per new consumer; doesn't scale with the service
  layer; the portability was never used.

### Merge env types into the domain types package (rejected)

- **Against:** mixes auto-generated infra types with hand-written domain shapes;
  forces `@cloudflare/workers-types` onto every consumer (incl. `ui`).

## Consequences

### Positive

- Simplified imports everywhere; zero drift (env regenerated from
  `wrangler.jsonc`); scales — a new `core` service that needs `db` is one import.

### Negative

- `db`/`auth` are Workers-only.
- **`GlobalProps` stripping.** `wrangler types` always emits a `GlobalProps`
  interface containing `typeof import("...server")` with a relative path to the
  app entry. When another package includes the generated file, TS follows that
  import and type-checks the app's server under the wrong module resolution. No
  wrangler flag excludes it, so the `cf-typegen` script strips it with a `node -e`
  post-process step (must be `node -e`, not `sed`, for cross-platform).

### Neutral

- Drizzle Studio uses its own config (`drizzle.config.local.ts`), bypassing the
  singleton — see [ADR-002](./002-d1-migrations.md).

## Implementation Notes

```
packages/env/
├── src/env.d.ts     # auto-generated by wrangler types (GlobalProps stripped)
├── src/index.ts     # comment-only; types are ambient from env.d.ts
└── package.json
```

The `cf-typegen` script (in `apps/web/package.json`):

```
wrangler types                                                  # full worker-configuration.d.ts for the app
wrangler types ../../packages/env/src/env.d.ts --include-runtime=false  # env-only types for the shared package
node -e "<strip GlobalProps>"                                   # remove GlobalProps to avoid cross-package resolution errors
```

> Never hand-edit `packages/env/src/env.d.ts` or the app's
> `worker-configuration.d.ts` — both are generated. Add bindings/vars to
> `wrangler.jsonc` / `.dev.vars` and re-run `pnpm cf-typegen`.

Consumer packages that use `cloudflare:workers` add `@workspace/env` +
`@cloudflare/workers-types` as devDeps and include
`../env/src/env.d.ts` in their tsconfig.
